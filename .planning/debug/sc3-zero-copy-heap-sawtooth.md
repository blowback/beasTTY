  implication: |

    object. This is by design of wasm-bindgen for `-> Vec<u8>` return types.

    (which use shared `wasm.memory.buffer`-backed views), NOT to the feed()
    return value path.

## Resolution

root_cause: |
  SC-3's "no per-frame allocation" expectation is violated by FIVE distinct
  per-Feed-click allocation sources. In order of likely byte-count contribution
  (largest first):

  1. **wasm-bindgen-generated `feed()` return-value `.slice()`**
     (www/pkg/bestialitty_core.js line 77).
     Caused by the Rust signature `pub fn feed(&mut self, bytes: &[u8]) -> Vec<u8>`
     in lib.rs line 70. wasm-bindgen MUST allocate a fresh JS Uint8Array with a
     new backing ArrayBuffer to copy the Vec<u8> contents out of wasm linear
     memory before freeing the wasm-side allocation. This happens on EVERY
     feed() call, regardless of whether the host_reply is empty (the common case).
     This is the most surprising contributor because it is invisible from
     www/main.js — it lives in the auto-generated bindings.

  2. **renderAscii string assembly** (www/main.js lines 42-59).
     Per click: one ~1944-char (~3888-byte) flat String allocation for the
     final flattened `out` assigned to textContent. Plus intermediate cons-strings
     from 1920 += operations (mostly GCable cheaply by V8).

  3. **parseHexEscapes Uint8Array construction** (www/main.js line 104).
     Per click: one fresh Uint8Array + backing ArrayBuffer (size = parsed byte
     count, typically <100 bytes for SC-3 input). Plus the JS Array `out` used
     internally.

  4. **reDeriveViews per-frame Uint8Array construction**
     (www/main.js lines 35-38, called from renderAscii line 44).
     Per click: TWO fresh Uint8Array header objects (~160 bytes total). The
     ArrayBuffer is shared (wasm.memory.buffer), but the wrapper objects are
     new. Plan 04 explicitly chose this defensive re-derivation over caching
     because it costs "microseconds"; SC-3 was written with the expectation
     that "no growing Uint8Array allocations" would mean zero allocations,
     not just zero significant byte-count, so the defensive re-derivation
     itself violates the literal SC-3 wording.

  5. **renderDirty Array.from + join** (www/main.js line 63).
     Per click: one 24-element JS Array + one 24-char String allocation.

  Combined: each Feed click allocates at MINIMUM 5+ distinct fresh JS objects
  totaling roughly 4-5 KB of fresh heap (dominated by the renderAscii string).
  GC reclaims these in batches → classic stair-case/sawtooth pattern that
  the user observed. The reported "Nodes" growth is most likely a Performance-
  timeline-sampling artifact of generalized heap pressure (no code path in
  www/main.js creates DOM nodes after init — only `.textContent = ` assignments).

  The "zero-copy Uint8Array views" promise of Phase 2 D-03 is correctly held
  ONLY for the grid pack-buffer READ path (gridView/dirtyView indexing into
  shared wasm.memory.buffer). It was never held for:
    (a) the feed() return value (Rust signature returns owned Vec<u8>),
    (b) the input byte path (parseHexEscapes → fresh Uint8Array → wasm copy),
    (c) the pre-text rendering path (1944-char string per frame),
    (d) the dirty bitmap stringification path (Array.from + join).

  SC-3 as written ("flat allocation profile after initial view construction,
  no growing heap sawtooth") is unsatisfiable without redesigning at least
  source #1 (the dominant contributor that the team was unaware of) and
  preferably #4 too. Sources #2, #3, #5 are inherent to the pre-text harness
  and become moot in Phase 3 (canvas renderer).

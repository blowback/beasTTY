# Phase 10 — UI Design Contract

> Visual and interaction contract for the SLIDE Receiver & Cancellation phase.
> Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Inherits from Phases 3, 4, 5, 6, 8, 9.** Phase 3 established the design
> system (CSS custom properties, two themes, top-bar chrome, focus-attribute
> idiom). Phase 4 added the `<details id="settings">` pane + the
> `mousedown preventDefault` focus-retention rule. Phase 5 added the inline
> error log. Phase 6 added the floating-chip pattern, the Settings-pane
> `.settings-row` idiom (`Clear scrollback` / `Auto connect on load` /
> `Reset all preferences`), and the `chunks: Uint8Array[]` + `Blob`
> session-log download anchor pattern. Phase 8 added the `Slide` wasm
> dispatcher and the `setWireOwner` TX-handoff contract (no UI surface).
> Phase 9 introduced the SLIDE visual language: the top-bar `[↑ Send file]`
> text-button, the `[data-drop-target]` 2px dashed-accent drop overlay,
> and the native `<dialog>` rewrite/rejection modal.
>
> **Phase 10's user-visible UI surface is intentionally small.** The phase
> is mostly architectural plumbing (Rust SM payload exposure, JS download
> dispatch, cancel state machine, IndexedDB integration). It introduces
> **one new chrome surface** and **one keyboard-contract update**:
>
> 1. A new Settings-pane row pulled forward from Phase 11's Settings
>    block: `[ ] Save received files to a folder` checkbox + adjacent
>    `[Choose folder…]` text-button + state hint string. Mirrors the
>    Phase 6 `.settings-row` idiom verbatim (CONTEXT D-02).
> 2. An update to the `keyboard.js` Esc-disambiguation chain inserting
>    SLIDE-cancel as new slot 2 of the existing
>    selection-cancel → paste-cancel → 0x1B chain (CONTEXT
>    §"Esc disambiguation slot"). No new visible element — keyboard
>    contract only.
>
> Phase 10 introduces **zero new typography sizes, zero new spacing
> tokens, zero new font-family declarations, zero new color tokens, zero
> new CSS attribute selectors, zero new keyboard chords, and no floating
> chip**. The chip surface (file count / percent / bytes / Cancel
> button / post-cancel summary) is the Phase 11 deliverable
> (SLIDE-25/26/27/28). Phase 10 progress is observable only via
> `window.__slide` introspection (Playwright tests).
>
> **Browser-owned chrome** appears during a recv session (showDirectoryPicker
> dialog, re-permission prompt, Chrome download bar, multi-download
> prompt). These are Chromium UI surfaces — BestialiTTY does NOT style or
> intercept them. They are documented in §"Browser-Owned Chrome" so the
> verifier and auditor know what to expect during real-hardware UAT but
> are NOT part of the BestialiTTY visual contract.

**`⚠` glyph color treatment (the only Phase 10 visual decision worth
calling out):** The U+26A0 warning sign renders in `var(--chrome-fg)` —
the same color as the surrounding state-string text. Rationale:

1. **No new color introduced.** Adding a yellow / amber accent for the
   `⚠` would inflate the design system for one use; adding a state-color
   token would pollute the Phase 5 amber/red/green Connect-button
   semantic.
2. **Glyph carries the warning.** `⚠` is a strong visual signal at
   12 px (the Hint size); it does not need additional color emphasis to
   be noticed.
3. **Calm tone.** Phase 6 invariant: "Calm, informational tone." A
   yellow `⚠` would over-signal what is a graceful-degradation hint, not
   an error. The user is informed; they can act when they want to.

### Settings-pane row — `Save received files to a folder` (CONTEXT D-02)

The row has **four runtime states** driven by `(prefs.slideRecvToFolder,
hasDirectoryHandle, permissionGranted)`:

| State | Trigger | Checkbox label (verbatim) | Adjacent button (verbatim) | State string (verbatim) |
|-------|---------|---------------------------|----------------------------|-------------------------|
| (a) Toggle off (default — first-ever visit) | `slideRecvToFolder === false` | `Save received files to a folder` | `Choose folder…` (button shown but inert until toggle is on — see Interaction Contracts) | `No folder selected` |
| (b) Toggle on, no folder yet | `slideRecvToFolder === true` && `getRecvDirHandle() === null` | `Save received files to a folder` | `Choose folder…` | `⚠ Pick a folder before next transfer` |
| (c) Toggle on, folder chosen, permission granted | `slideRecvToFolder === true` && handle present && `requestPermission` resolved `'granted'` | `Save received files to a folder` | `Change folder…` | `Saving to: {dirPath}` (e.g. `Saving to: MicroBeast`) |
| (d) Toggle on, folder chosen, permission denied / pending | `slideRecvToFolder === true` && handle present && `requestPermission` resolved `'prompt'` or `'denied'` | `Save received files to a folder` | `Re-allow folder…` | `⚠ Permission needed for {dirPath}` |

**Verbatim formatting:**

```
[ ] Save received files to a folder
     [Choose folder…]   No folder selected
     Received files land in your Downloads folder. Toggle this to pick a fixed destination.
```

```
[x] Save received files to a folder
     [Change folder…]   Saving to: MicroBeast
     Received files are written here directly. Toggle off to revert to your Downloads folder.
```

```
[x] Save received files to a folder
     [Choose folder…]   ⚠ Pick a folder before next transfer
     Received files are written here directly. Toggle off to revert to your Downloads folder.
```

```
[x] Save received files to a folder
     [Re-allow folder…] ⚠ Permission needed for MicroBeast
     Received files are written here directly. Toggle off to revert to your Downloads folder.
```

**Notes:**

- Ellipsis `…` is **U+2026 HORIZONTAL ELLIPSIS** in `Choose folder…` /
  `Change folder…` / `Re-allow folder…`. Not three ASCII dots. Matches
  Phase 5 `Connecting…` and Phase 9 `(sending…)` conventions.
- Warning glyph is **U+26A0 WARNING SIGN** `⚠` followed by a single
  ASCII space then the warning text. Not the variant emoji `⚠️` (with
  U+FE0F selector — that triggers emoji rendering on some platforms),
  not the heavy form `⚠️`. Bare U+26A0 only.
- The `{dirPath}` placeholder in states (c) and (d) is the
  `FileSystemDirectoryHandle.name` property — typically the leaf folder
  name only (e.g. `MicroBeast`, not `~/Downloads/MicroBeast`). The Web
  File System Access API does not expose the full path for privacy; the
  `~/Downloads/MicroBeast` form shown in 10-CONTEXT.md §Specifics
  Settings-pane row layout is illustrative only. **Spec lock:** state
  strings use `handle.name` verbatim. If a future browser provides
  `handle.queryFullPath()` or similar, the executor may upgrade with
  no spec change.
- "Pick a folder" / "previously-chosen folder" / "Re-allow" — vocabulary
  chosen so a daily-driver user understands the state without needing
  to know the File System Access API permission model.
- The inline hint paragraph **swaps** with toggle state (off vs on
  variants are different copy). The state string `<span>` swaps among
  the four runtime variants.

### Esc key behavior change (CONTEXT §"Esc disambiguation slot")

**No new visible element.** Phase 10 inserts SLIDE-cancel as new slot 2
in the existing `keyboard.js:202-227` Esc disambiguation chain. The
chain becomes (priority order):

1. `Esc` while selection-dragging → cancel selection drag (existing,
   Phase 6)
2. **`Esc` while `slide.isActive('recv')` → cancel SLIDE recv session
   (NEW Phase 10)** — invokes the cancel sequence in
   §"Interaction Contracts" below
3. `Esc` while paste-pump active → cancel paste (existing, Phase 5
   D-18; was slot 2 pre-Phase 10)
4. `Esc` fallthrough → encode 0x1B and forward to remote (existing,
   Phase 4; was slot 3 pre-Phase 10)

**User-visible effect:** Pressing Esc during an active recv session
terminates the session. There is no toast, no chip, no banner — the
in-canvas grid keeps rendering whatever the Z80 sends after the
cancel handshake completes. Phase 11 SLIDE-28 attaches a 5-second
"Cancelled — N of M files transferred" auto-hide chip.

**No new keyboard chord. No new modifier intercept. No new shortcut
in `<details class="reserved">` Browser-reserved Ctrl combinations
list.** The chord is plain `Esc`, which is already documented (Phase 4
encodes it as 0x1B; Phase 5 added paste-cancel; Phase 6 added
selection-cancel; Phase 10 adds SLIDE-cancel).

### Empty / error / destructive states

| State | Copy / behavior |
|-------|----------------|
| Empty state | n/a — Phase 10 has no list-of-items surface. The Settings row is always rendered (visible in the Settings pane); the in-canvas SLIDE recv session has no UI list. |
| Error state — directory permission denied on reload | State (d): `⚠ Permission needed for {dirPath}` + button label `Re-allow folder…`. Toggle stays on. Per CONTEXT D-04, picker dismissal during a session falls back to anchor-click for the remainder of that session; toggle stays on; no re-prompt. **No additional error surface — the state string IS the error surface.** |
| Error state — picker dismissal during recv session | Per CONTEXT D-04: silently fall through to anchor-click for the rest of the session. Toggle stays on. The Settings row state string is NOT updated mid-session (would race against the chip — Phase 11 owns mid-session UI). |
| Error state — `~999` collision retry exhausted | Per CONTEXT D-06: fall through to anchor-click for the offending file; subsequent files in the batch continue attempting the directory path normally. **No user-visible UI surface in Phase 10.** Phase 11 chip will show "saved as MY-FILE.TXT in Downloads" or similar; Phase 10 ships console-only. |
| Error state — IndexedDB unavailable (incognito mode without storage) | Toggle reads as off on every page load; user can still toggle on for the current session but the handle is not persisted. **No user-visible error surface in Phase 10.** Falls through to console.warn (matches Phase 6 prefs `QuotaExceededError` precedent). |

| Destructive confirmation | n/a — no destructive actions in the Phase 10 chrome. The Esc-cancel is the closest analog, but it acts on an in-flight transfer, not on persistent data. ### Copywriting invariants (inherited from Phases 5 + 6 + 9, preserved here)

- **No emojis anywhere.** Pure ASCII / English (Unicode `…` for
  ellipses, `⚠` for the warning sign, `~` for the collision suffix
  numeric — `~1`, `~2`, etc. — are typography punctuation / glyphs,
  not pictorial emoji).
- **No exclamation marks.** Calm, informational tone.
- **No "please" in UI strings.**
- **No "Loading…" or "Receiving…" spinners.** Phase 10 has no UI
  surface for in-flight progress; the Phase 11 chip lands progress
  display.
- **No modal dialogs.** Inherited from Phase 6 + narrowed by Phase 9
  (`<dialog>` is the rewrite/rejection confirm only). Phase 10 ships
  zero modals — picker / re-permission dialogs are browser-owned and
  not BestialiTTY's responsibility.
- **No native browser confirms (`window.confirm`).** Same as Phase 6.
- **State strings end without trailing punctuation** (no period,
  no exclamation). Hint paragraphs end with a period. Tooltips end
  without punctuation.
- **Filename literals shown verbatim** — handle name, e.g. `MicroBeast`,
  is whatever Chromium provides via `handle.name`. Never lowercased,
  never decorated.
- **Locked verbatim strings:** the table above is the source of truth.
  No localization, no i18n.

## Interaction & State Contracts

These are the load-bearing interaction rules Phase 10 MUST implement.
Most are invisible to the user but critical to phase success.

### Settings-row state machine

| Current state | Trigger | Next state | Side effects |
|---------------|---------|------------|--------------|
| (a) toggle off, no folder | click `<input id="slide-recv-to-folder-checkbox">` (toggle on) | (b) toggle on, no folder | Set `prefs.slideRecvToFolder = true` (debounced via existing `savePrefs` Phase 6 contract); enable `[Choose folder…]` button (remove `disabled` attribute); update state string to `⚠ Pick a folder before next transfer`; swap inline-hint paragraph copy from "Received files land…" to "Received files are written…" variant |
| (b) toggle on, no folder | click `[Choose folder…]` | (b → c) await picker | Browser shows `showDirectoryPicker()` dialog. Picker resolves → save handle to IndexedDB via `setRecvDirHandle(handle)`, request readwrite permission, transition to (c). Picker rejects (user cancels) → no state change, button label unchanged, state string unchanged |
| (b) toggle on, no folder | click `<input>` (toggle off) | (a) toggle off, no folder | Set `prefs.slideRecvToFolder = false`; disable `[Choose folder…]` button; update state string to `No folder selected`; swap inline-hint paragraph back to "Received files land…" variant |
| (c) toggle on, folder chosen, granted | click `[Change folder…]` | (c → c') await picker | Browser shows `showDirectoryPicker()`; resolves → replace handle in IndexedDB, request permission, stay in (c) with new path; rejects → no state change |
| (c) toggle on, folder chosen, granted | click `<input>` (toggle off) | (a) toggle off, no folder | Set `prefs.slideRecvToFolder = false`; the IndexedDB handle is **kept** (not cleared) — re-toggling on restores state (c) without re-picking. State string updates to `No folder selected`. Per CONTEXT D-03 — handle persists across reloads + across toggle off/on cycles within a session |
| (d) toggle on, folder chosen, denied | click `[Re-allow folder…]` | (d → c on grant; d → d on deny) | Call `handle.requestPermission({ mode: 'readwrite' })`; on `'granted'` resolve → transition to (c); on `'prompt'` / `'denied'` resolve → stay in (d) |
| (d) toggle on, folder chosen, denied | click `<input>` (toggle off) | (a) toggle off, no folder | Same as (c) → (a). Handle preserved. |
| Page load with `prefs.slideRecvToFolder === true` AND `getRecvDirHandle() !== null` | boot | call `handle.requestPermission()` → on grant: (c); on prompt/deny: (d) | The re-permission prompt appears at boot if the toggle was on and a handle exists. Per CONTEXT D-03: Chrome shows a one-click "Allow" dialog for previously-granted handles. |
| Page load with `prefs.slideRecvToFolder === true` AND `getRecvDirHandle() === null` | boot | (b) toggle on, no folder | Toggle is on but handle is gone (e.g. user manually cleared site data). State string `⚠ Pick a folder before next transfer`. |
| Page load with `prefs.slideRecvToFolder === false` | boot | (a) toggle off, no folder | Default first-visit state. |

**Ordering note:** the `<input>` (checkbox) state and the
`[Choose folder…]` button state are decoupled — a user can toggle
on/off without clicking the button, and the button is inert when the
toggle is off (per state (a)). The button only fires when the user
explicitly clicks it.

### Active recv session — visible-element contract

During an active recv session (post-`ESC^SLIDE`-wakeup,
pre-cancel-or-completion), Phase 10 surfaces **zero visible UI
elements**. Specifically:

| Surface | Phase 10 behavior |
|---------|------------------|
| Canvas (`#terminal`) | **Suspended.** Per Phase 8 setWireOwner('slide') contract, the terminal parser is paused; no new bytes paint. The canvas continues to render the last frame painted before wakeup (frozen image). |
| Top-bar | **Unchanged.** No `[↓ Receive]` button, no `[↓ Cancel]` button. The `[↑ Send file]` button (Phase 9) **is** disabled-with-suffix-label `↑ Send file (sending…)` because the SLIDE wire-owner check applies symmetrically — see Phase 9 D-15 + D-18 (transfer in progress, regardless of direction, gates the send button). Phase 10 verifies this is already covered by the Phase 9 logic; if not, planner extends `file-source.js` to gate on `slide.mode !== 'terminal'`. |
| Settings row state string | **Unchanged during the session.** Per CONTEXT D-04: picker dismissal mid-session falls through to anchor-click without re-prompting. The Settings row reflects the **persistent toggle state**, not the **session state**. Mid-session UI is Phase 11 chip surface. |
| Floating chip area (bottom-left of `#terminal-wrapper`) | **Empty.** Phase 11 lands the chip. |
| Browser download bar | **Variable.** When toggle is OFF and a file completes, Chrome's bottom-edge download bar shows the filename briefly. When toggle is ON and permission is granted, no bar — the file lands silently in the chosen folder. This is browser-owned. |

**The user observes a recv session via:**

1. The frozen canvas (visual signal: "something is happening, terminal
   is paused")
2. The Chrome download bar (per file, if toggle is OFF)
3. The arrival of files in their Downloads folder OR chosen folder
4. Console logs (`window.__slide` introspection — Playwright tests + dev
   tools)

This minimal-chrome posture is **deliberate** per CONTEXT §"Cancel UX
in Phase 10 (pre-chip)" — Phase 11 lands the chip surface to
fix the discoverability gap.

### Esc-cancel contract (CONTEXT §"Esc disambiguation slot")

The Esc key has FOUR possible behaviors at runtime, in priority order
(extends Phase 6 §"Esc key disambiguation"):

1. **If `selection.isInFlight`** (`dragging` state): cancel selection,
   return — Phase 6 D-19. Esc does NOT propagate to remote.
2. **Else if `slide.isActive('recv')`**: cancel SLIDE recv session, return
   — Phase 10 NEW. Esc does NOT propagate to remote, does NOT reach the
   paste-pump or 0x1B paths.
3. **Else if `pastePump.isActive()`**: cancel paste, return — Phase 5
   D-18. Was slot 2 pre-Phase 10.
4. **Else (default)**: Phase 4 path — encode 0x1B (ESC), send to
   remote.

**Slot 2 SLIDE-recv-cancel side effects** (per CONTEXT §"Cancel timing
windows"):

| Step | Action | Timing |
|------|--------|--------|
| 1 | Settle in-flight `writer.write` calls | `Promise.allSettled(inflightWrites)` capped at 200 ms |
| 2 | Push CTRL_CAN to outbound (single byte 0x18) | `slide.cancel()` then `writeSlideFrame` (existing Phase 8 fire-and-forget path) |
| 3 | Wait for Z80 CTRL_CAN echo | up to 500 ms; SM in CancelPending silent-drain state |
| 4 | Drain post-echo | 100 ms `setTimeout` |
| 5 | If echo never arrived (stock slide.com without v0.2.1 amendment), force_idle escape hatch | `slide.force_idle()` per ADR-003 §3 |
| 6 | Re-arm — exit recv mode, owner back to terminal | `txSinkRef.setWireOwner('terminal')`; `mode = 'terminal'`; bell-overlay / scrollback / cursor blink resume |

**Total cancel duration:** 200 + 500 + 100 = 800 ms typical;
2000 ms absolute timeout per ADR-003 §3 (`Promise.race` wraps the
whole sequence).

**User-visible during cancel:** none in Phase 10. The canvas remains
frozen until step 6 unfreezes it; then normal terminal output resumes.
Phase 11 SLIDE-28 attaches the 5-second post-cancel "Cancelled — N of
M files transferred" auto-hide chip.

### Toggle persistence (CONTEXT D-02 + D-03)

| Surface | Storage | Persistence |
|---------|---------|-------------|
| `slideRecvToFolder: false` boolean | `bestialitty.prefs` localStorage blob (Phase 6 versioned schema) | Survives reload |
| `slideRecvDirectoryHandle` | IndexedDB store `bestialitty-handles`, key `recv_directory` (planner finalizes exact names) | Survives reload; Chrome shows one-click Allow on re-permission |
| Schema version bump | `bestialitty.prefs.version` should increment if `slideRecvToFolder` is added as a new field | Planner picks: bump from 1 to 2 OR rely on the Phase 6 defensive merge `cached = { ...DEFAULTS, ...parsed, ... }` to fill the missing field with `false` (the latter requires no migration code; the former is more explicit). Researcher recommends the latter for simplicity (matches Phase 6's existing field-additive pattern) |

### Multi-file batch download — visible behavior

Per CONTEXT D-01 — toggle state determines path, file count does not:

| Mode | File count | User-visible chrome |
|------|-----------|---------------------|
| Toggle OFF | 1 | Chrome download bar shows the file briefly. |
| Toggle OFF | N > 1 | Chrome download bar shows each file in turn (250 ms inter-file gap per CONTEXT C-01). On the second file, Chrome may show a "Allow {site} to download multiple files?" prompt; if user clicks Block, all subsequent files in the batch silently fail. **Phase 10 does NOT detect this — failed files are dropped without user notification.** Phase 11 chip will track per-file success/failure. |
| Toggle ON, granted | 1 | No browser chrome. File lands silently in chosen folder. |
| Toggle ON, granted | N > 1 | No browser chrome. All files land silently in chosen folder. **No "multiple downloads" prompt** because the showDirectoryPicker grant is per-folder, not per-file. |
| Toggle ON, denied / picker dismissed | any | Falls through to anchor-click for the rest of the session (per CONTEXT D-04). User sees Chrome download bar per file, and may see the "Allow multiple downloads?" prompt. |

### Chrome multi-download prompt — design acknowledgment

Per PITFALLS §10: when the toggle is OFF and the user receives a
multi-file batch, Chrome may show a "Allow {site} to download multiple
files?" prompt. If the user clicks **Block**, all subsequent files in
that batch (and that origin) silently fail until the page is reloaded.
**Phase 10 cannot detect this.** This is a known limitation; the chip
surface in Phase 11 will track per-file `download.failed` events
(if any) and surface the failure count.

The 250 ms inter-file gap (CONTEXT C-01) is the mitigation: by spacing
downloads, BestialiTTY signals to Chrome that this is intentional
(not a malicious flood) and reduces the likelihood of the prompt.

## Accessibility Contract

### Focus order (Tab sequence — Phase 10 additions only)

The Phase 10 additions slot into the existing Phase 9 Tab order at the
end of the Settings pane. Starting from `#reset-prefs-button`
(last Phase 6 row):

1. `#reset-prefs-button` (Phase 6 — last entry)
2. **`#slide-recv-to-folder-checkbox` (Phase 10 NEW)**
3. **`#slide-recv-folder-button` (Phase 10 NEW)** — only Tab-focusable
   when the checkbox is checked AND button is not `disabled`
4. `<summary>` of `<details id="debug">` (Phase 3)

**Rationale:** Standard inline DOM order. The `<input>` checkbox comes
before its associated button because the button's enabled state depends
on the checkbox state — Tab-traversing the checkbox first allows a
keyboard user to toggle on, then Tab to the button to pick a folder.

### Focus retention on Phase 10 control click

| Trigger | Element | Behaviour |
|---------|---------|-----------|
| `mousedown` | `#slide-recv-to-folder-checkbox` | `e.preventDefault()` to retain terminal focus; native `<input>` checkbox toggles via the click handler. **Confirmed implementation: rely on native click toggle; do NOT pre-flip in mousedown.** Matches Phase 6 D-34 + Phase 4 Plan 04-04 Rule 1 fix precedent. |
| `mousedown` | `#slide-recv-folder-button` | `e.preventDefault()` to retain terminal focus; click handler fires `showDirectoryPicker()` async. The picker dialog itself takes focus (browser-owned); on dialog close, focus returns to `#terminal-wrapper`. |

**Picker-dialog focus return:** when `showDirectoryPicker` resolves OR
rejects, Phase 10 explicitly calls `terminalWrapperEl.focus()` to
restore the daily-driver focus invariant. Same pattern as Phase 9
`<dialog>` close (file-source.js).

### Screen reader / aria

| Element | aria treatment |
|---------|----------------|
| `<input type="checkbox" id="slide-recv-to-folder-checkbox">` | Native `<input>` with associated `<label>` — UA exposes "Save received files to a folder, checkbox, checked / not checked" |
| `<button id="slide-recv-folder-button">` | Native `<button>` — UA exposes "Choose folder, button, dimmed / pressed". The disabled state when toggle is off is conveyed via the native `disabled` attribute |
| `<span id="slide-recv-folder-status" class="hint">` | Plain span — readable in normal browse mode. **No `aria-live`** in Phase 10 (state string only changes on user interaction with the row, not from background events); Phase 11 chip introduces `aria-live="polite"` for session progress |
| `<p class="hint">` paragraphs | Plain text — readable inline |
| `⚠` glyph in state strings | Renders as "warning sign" in screen readers per Unicode CLDR. Acceptable — the glyph IS the warning; SR users get the same signal as sighted users. No `aria-label` override needed |

**No new `role` attributes, no new `aria-*` attributes.** Phase 10
ships with native semantics only.

### Keyboard-only operation

| Action | Key sequence |
|--------|--------------|
| Toggle "Save received files to a folder" on | `Tab` to `#settings` summary → `Enter` to expand → `Tab` repeatedly until `#slide-recv-to-folder-checkbox` is focused → `Space` to toggle |

### Color-blindness considerations

The `⚠` glyph is the sole visual signal in the warning state strings;
color is not load-bearing (the glyph is rendered in the same color as
the surrounding text). Any monochrome or color-blind user sees the same
glyph, same contrast, same warning. Verified.

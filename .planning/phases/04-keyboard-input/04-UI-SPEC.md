# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4 (Keyboard Input). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inherits from Phase 3.** Phase 3 established the full design system (two themes, CSS custom properties, JetBrains Mono + hand-drawn bitmap font, top-bar chrome, Debug `<details>` pane). Phase 4 adds a **Settings `<details>` pane** and **TX-byte hex strip inside the existing Debug pane** — both reuse Phase 3 tokens verbatim. No new color tokens, no new font-family declarations.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla ES modules — no React, no bundler, no component library) |
| Preset | not applicable |
| Component library | none (native `<details>` / `<fieldset>` / `<input>` / `<button>`) |
| Icon library | none (Phase 4 surfaces are text-only — no icons needed) |
| Font (clean theme chrome) | JetBrains Mono Regular (self-hosted WOFF2, `font-display: block`) — falls back to `ui-monospace, SFMono-Regular, Menlo, Consolas, monospace` |
| Font (CRT theme chrome) | Same CSS stack — CRT theme only swaps colors via `[data-theme="crt"]`; Phase 4 UI chrome shares the clean-theme monospace stack (the 8×16 bitmap font applies only to canvas glyphs, not DOM chrome — Phase 3 D-01) |
| Framework | Framework-free — vanilla DOM, native `<details>`/`<fieldset>`/`<input type="radio">`/`<input type="checkbox">` |

**Rationale:** Phase 3 shipped a fully wired design system. Phase 4 extends the existing Debug `<details>` styling block to a sibling Settings `<details>` with identical CSS (padding, border, summary font-size, hint color, etc.). Zero net-new tokens.

## Typography

Phase 4 reuses Phase 3's established typography. Sizes: 3 distinct values (14px, 13px, 12px). Weights: 2 distinct values (regular 400, semibold 600). Line-height: 1.4 (matches Phase 3 `body`).

| Role | Size | Weight | Line Height | Usage in Phase 4 |
|------|------|--------|-------------|------------------|
| Body | 14px | 400 | 1.4 | Chrome default — body background, top-bar buttons, `<details>` summary text (Settings + Debug summaries) |
| Label | 13px | 400 | 1.4 | Settings-pane `<label>` text next to local-echo checkbox; `<legend>` text ("Enter key sends"); individual radio `<label>` text (cr / lf / crlf) |
| Hint | 12px | 400 | 1.4 | Hint lines under checkbox and under radio group; the browser-reserved `<details>` note body text; the TX hex strip itself |
| TX strip (monospace) | 12px | 400 | 1.4 | `<pre id="tx-strip">` hex-byte output — inherits font-family from parent `<pre>` default (monospace) which resolves to the same ui-monospace stack as the rest of chrome |
| Button (Reset TX) | 12px | 400 | 1.4 | Matches existing Debug pane button size (see `#debug button` rule in index.html line 152-159) |

**Rationale for 13px Label:** Phase 3's existing Debug pane uses 14px for summary and 12px for body/hint/button. Phase 4's Settings labels sit between those — 13px gives visual hierarchy: summary (14) > label (13) > hint (12). This single new size is the ONLY typography addition beyond Phase 3, and it is a single step in the existing scale, not a new scale dimension.

**Decision point if 13px is rejected by checker:** Collapse Label to 12px (match hint) — acceptable fallback. Must NOT introduce any size above 14px (no H2/H3 in Phase 4 — a pane has one summary line and is read top-to-bottom, no heading hierarchy needed).

### Format of TX hex strip content

- Space-separated two-digit uppercase hex pairs, newest on the right.
- Example after pressing ArrowUp then ArrowDown: `1B 41 1B 42`
- Max 64 bytes shown (ring buffer holds up to 1024, strip displays last 64 — confirms CONTEXT D-07 "last 64 bytes" and CONTEXT D-15 "shows last 64 bytes").
- Empty state (before any keys pressed): shows the placeholder string above, not an empty `<pre>` (an empty `<pre>` would collapse the Debug pane's vertical rhythm).

### Copywriting invariants

- **No emojis anywhere.** Pure ASCII / English.
- **No exclamation marks.** Calm, informational tone — this is a daily-driver diagnostic, not a promotional UI.
- **Byte values always in hex with `0x` prefix when written inline in prose** (e.g. "0x0D") and as **two-digit uppercase hex pairs WITHOUT `0x` prefix** when displayed in the TX strip `<pre>` (e.g. `1B 41`). Rationale: inline prose needs the prefix to disambiguate from decimal; the `<pre>` strip is understood to be hex by context and cleaner without prefixes (matches how a serial-debugger shows packet hex).
- **Control-code names:** use `Ctrl+X` with plus sign (matches Chromium's own chord docs). Never `CTRL-X` / `^X` / `⌃X`.

### Primary CTA (phase-level)

| Element | Copy |
|---------|------|
| Primary CTA | Phase 4 has **no phase-level primary CTA button**. The primary interaction is passive — **type on the canvas** and keys are forwarded. The Settings pane toggles default to the zero-config path (local-echo OFF, CR mode) so the user needs to do nothing to validate the phase. Reset TX is a secondary diagnostic action, not the primary CTA. |

### Empty / error / destructive states

| Element | Copy |
|---------|------|
| Empty state heading | **not applicable** — Phase 4 surfaces are always populated: Settings pane has two static controls and a note; Debug TX strip has a placeholder string for its empty state (see above). There is no list, feed, or collection that could be "empty". |
| Empty state body | TX strip placeholder: `(none yet — press any key on the terminal to see TX bytes)` (see above). |
| Error state | **not applicable** — Phase 4 has no failure modes that surface to the DOM. A key that isn't in the `KeyCode` enum is a silent drop (CONTEXT D-17), not an error. IME double-emit is a silent guard (CONTEXT D-06), not an error. The wasm `encode_key_raw` has no error arm (returns `Vec<u8>`, never throws in JS). Console logs exist for diagnostics but are not user-facing. |
| Destructive confirmation | **not applicable** — Reset TX is non-destructive (clears a diagnostic display, no persisted state, no undo required). Local-echo and CR/LF toggles are non-destructive (flip at will; no data loss). No confirmation dialog. |

## Interaction Contracts

These are the load-bearing interaction rules Phase 4 MUST implement. They are mostly invisible to the user but critical to phase success.

### Focus retention on toolbar click (CONTEXT D-16)

| Trigger | Element | Behaviour | Why |
|---------|---------|-----------|-----|
| `mousedown` | `#theme-toggle` | `e.preventDefault()` | Prevents native focus transfer — `#terminal-wrapper` keeps focus |
| `mousedown` | each `#phosphor-group button[data-phosphor]` (3 buttons) | `e.preventDefault()` | Same |
| `mousedown` | `#local-echo` (checkbox) | `e.preventDefault()` on the checkbox's `mousedown`, but the click handler still toggles the checkbox programmatically via `setLocalEcho(!checkbox.checked)` | Native `mousedown` preventDefault suppresses focus AND the click → check toggle; we restore the toggle explicitly to preserve interaction |
| `mousedown` | each CR/LF radio (`#crlf-cr`, `#crlf-lf`, `#crlf-crlf`) | `e.preventDefault()` + explicit `radio.checked = true` + call `setCrlfMode(value)` | Same pattern as checkbox |
| `mousedown` | `#tx-reset` button | `e.preventDefault()` + call `resetTx()` from tx-sink.js | Focus retained; button action fires |
| **Keyboard activation** (Tab-to-element then Space/Enter) | All above | **NO preventDefault** — native click path fires, state mutates normally | CONTEXT D-16: "Keyboard activation of these same buttons (Tab-to, Space) still works because `mousedown` doesn't fire on keyboard activation." This is a platform truth, not a Phase 4 decision — but the planner and executor MUST NOT add a workaround. |

**Verification:** After clicking any of the above with a mouse, `document.activeElement === document.getElementById('terminal-wrapper')` MUST be true.

### Keydown delegation order (CONTEXT D-01, D-03)

On `#terminal-wrapper`, two keydown listeners exist after Phase 4 wiring:

1. **Phase 3's listener** (`www/renderer/chrome.js` line 76) — handles `Ctrl+Alt+T`, `Ctrl+{+/-/0}` — calls `e.preventDefault()` synchronously then returns.
2. **Phase 4's listener** (`www/input/keyboard.js` new) — attached AFTER Phase 3's (ordering by main.js wiring sequence). Runs every keydown. First check: `if (e.defaultPrevented) return;` — skips any key Phase 3 already handled. Then runs the Phase 4 decision tree (IME guard → control key → printable → silent drop).

**Consequence:** exactly one `preventDefault()` call per forwarded key. No double-handling. No stolen shortcuts.

### Unhandled keys pass through (CONTEXT D-17)

The following keys MUST NOT be `preventDefault`-ed. They pass through to the browser, preserving daily-driver utility:

- F1–F12 (devtools, reload, fullscreen)
- Home, End, PgUp, PgDn (browser scrolling)
- Insert, Delete
- PrintScreen, CapsLock, ScrollLock, NumLock
- ContextMenu
- Meta / Windows key alone
- Any key combination Phase 3 already claimed (Ctrl+Alt+T, Ctrl+Equal, Ctrl+Minus, Ctrl+Digit0)

### Composition / IME (CONTEXT D-06)

- `compositionstart` on `#terminal-wrapper` sets internal `isComposing` flag → keydown early-returns
- `compositionupdate` is a no-op
- `compositionend` iterates `event.data` bytes and emits each via `encode_key_raw`; clears flag
- `keydown` ALSO checks `e.isComposing` as belt-and-braces (defensive against Chromium version drift)

Result: pressing any CJK IME sequence on the canvas emits the composed result **once**, not twice. No visible UI surface for IME state — this is an invisible correctness contract.

### TX strip refresh (CONTEXT D-15)

- Observer pattern: `www/input/tx-sink.js` exports `registerTxObserver(fn)` and calls every registered observer on each `pushTxBytes(bytes)`.
- The Settings/Debug wiring in main.js registers one observer that calls `txStripEl.textContent = formatHexStrip(64)` on each push.
- **No polling.** No `setInterval`. Update is synchronous-after-push, not tied to rAF.
- When Debug pane is collapsed, the `<pre>` still updates — CSS `display: none` on the parent `<details>` when closed doesn't stop `textContent` writes; opening the pane shows current state immediately. Matches CONTEXT D-15: "hidden visually (not removed) when Settings is collapsed, but always updated".

### Reset TX behaviour

- `<button id="tx-reset">` click → `resetTx()` in tx-sink.js → zeroes the ring buffer + notifies observers.
- TX strip `<pre>` updates to show the empty-state placeholder string again (`(none yet — press any key on the terminal to see TX bytes)`).
- No confirmation. No undo. Non-destructive (no persisted state).

### Settings pane state (in-memory only)

- `#local-echo` checkbox state lives in `keyboard.js` module scope, default `false`.
- CR/LF radio group state lives in `keyboard.js` module scope, default `'cr'`.
- Page reload resets both. **No localStorage** — Phase 6 (PREF-02) adds persistence.

## Accessibility Contract

### Focus order (Tab sequence)

Starting from a freshly loaded page with focus on `#terminal-wrapper`:

1. `#terminal-wrapper` (default focus at boot — `chrome.js` calls `.focus()` at end of `wireChrome`)
2. `#theme-toggle` (Phase 3)
3. `#phosphor-group` buttons (Phase 3 — 3 buttons)
4. `<summary>` of `<details id="settings">` (expands pane on Space/Enter)
5. If expanded: `#local-echo` checkbox
6. If expanded: `#crlf-cr` radio (arrow keys move within group per native radio semantics)
7. If expanded: `<summary>` of inner `<details class="reserved">` (expands inner note)
8. `<summary>` of `<details id="debug">` (Phase 3)
9. If expanded: `#input` textarea (Phase 3)
10. If expanded: `#feed`, `#stress64k` (Phase 3)
11. If expanded: `#tx-reset` (Phase 4 — tabbable after `#stress64k`)

**Rationale:** Native tab order via DOM order. No `tabindex` beyond Phase 3's existing `#terminal-wrapper tabindex="0"` and `#terminal tabindex="-1"`. No custom focus traps.

### Semantic markup

- `<details>` + `<summary>` for disclosure — native, no ARIA needed.
- `<fieldset>` + `<legend>` for the CR/LF radio group — native, no `role="radiogroup"` override needed (unlike Phase 3's phosphor buttons, which use `role="radiogroup"` because they're `<button>` elements, not `<input type="radio">`).
- `<label for="...">` for every form control — native label binding.
- `<input type="checkbox">` and `<input type="radio">` — native controls; Chromium renders accessible names from the associated `<label>`.
- No `aria-label` / `aria-describedby` needed — labels and legends do the accessibility work.

### Screen reader note

PROJECT.md Out-of-Scope lists "Screen-reader support on canvas" as deferred. The Phase 4 Settings chrome IS accessible (standard DOM with standard labels), but the canvas terminal content is not. This asymmetry is intentional and documented.

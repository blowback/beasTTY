# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5 (Web Serial Transport). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inherits from Phase 3 + Phase 4.** Phase 3 established the design system (two themes, CSS custom properties, JetBrains Mono + bitmap font, top-bar chrome, Debug `<details>` pane). Phase 4 added the Settings `<details>` pane pattern and the TX hex strip. Phase 5 adds **four surfaces** that all reuse Phase 3/4 tokens verbatim:
>
> 1. A stateful top-bar `#connect-button` (new DOM, existing top-bar styling)
> 2. A new `<details id="connection">` pane between top-bar and terminal-wrapper (mirrors `#settings` CSS)
> 3. A new `<button id="paste-test">` inside the existing Debug pane (reuses `#debug button` styling)
> 4. A polite-fail full-page takeover that MUST NOT depend on theme/phosphor tokens (they haven't loaded yet)
>
> Phase 5 introduces **two new color hex values** (amber `#e0b030` and red `#e04040` for Connect button state-borders) and **zero new typography sizes, zero new spacing tokens, zero new font-family declarations**.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla ES modules — no React, no bundler, no component library; PROJECT.md + Phase 2 D-14 lock this posture) |
| Preset | not applicable |
| Component library | none (native `<details>` / `<fieldset>` / `<input>` / `<button>` / `<select>`) |
| Icon library | none (Phase 5 surfaces are text-only; state signaling is via button border color, not icons) |
| Font (chrome, both themes) | JetBrains Mono Regular self-hosted WOFF2 + fallback chain `ui-monospace, SFMono-Regular, Menlo, Consolas, monospace` — inherited verbatim from Phase 3 D-02 / Phase 4 UI-SPEC; Phase 5 adds no new font-family declarations |
| Font (polite-fail page) | `system-ui, -apple-system, Segoe UI, sans-serif` — **intentionally different**: the polite-fail page renders before wasm + JetBrains Mono load, so it uses platform UI sans-serif to avoid a FOIT. The message is read once, not stared at daily. |
| Framework | Framework-free — vanilla DOM, native form controls, CSS custom-property driven themes |

**Rationale:** Phase 3 + Phase 4 shipped the whole design system. Phase 5 is the first phase that introduces a UI with "non-default" visual states (connecting/connected/port-lost) — these states ride the existing `--chrome-*` token plumbing via a new `[data-state]` attribute on a single button, and a new Connection pane that is CSS-identical to the Settings pane.

## Spacing Scale

Phase 5 reuses Phase 3 + Phase 4's established scale. All values are multiples of 4.

| Token | Value | Usage in Phase 5 |
|-------|-------|------------------|
| xs | 4px | Connect-button internal padding (vertical); error-log entry vertical margin; paste-progress row vertical padding; Connection pane hint-line vertical margin |
| sm | 8px | Connection pane internal padding (horizontal + vertical); Connect-button internal padding (horizontal, slightly tighter than other top-bar buttons — see exception below); serial-config fieldset padding; inline error-log outer padding |
| md | 16px | Connection pane outer vertical margin (matches `#settings` / `#debug` `margin: 16px auto`); serial-config fieldset flex-gap between rows |
| lg | 24px | (unused in Phase 5) |
| xl | 32px | (unused in Phase 5) |

**Exceptions:**

| Exception | Value | Rationale |
|-----------|-------|-----------|
| Connect button horizontal padding | `4px 10px` (10px, not multiple of 4) | Research Pattern 5 Example CSS codifies this — the Connect button label rotates through `Connect` (7 chars) / `Connecting…` (11 chars) / `Disconnect` (10 chars) / `Reconnecting…` (13 chars) / `Reconnect` (9 chars); 10px horizontal padding visually balances the 13-char widest label without forcing the top-bar to reflow when the label shortens. The other top-bar buttons (theme-toggle + phosphor radio group) use 8px horizontal padding; a 10px one-off on this single button is acceptable given the label-width variance. Treat 10px as the sole compliant deviation, not a new scale token. |
| Connect button border width | 1px (consistent) across all 5 states | State signal is border **color**, never border **width** — swapping width between states would cause layout reflow on every state change, visible on any top-bar alignment. |
| Inline error log entry vertical gap | 2px between entries | Error log holds **5 entries max** in a tight vertical stack; 4px gaps would push the Connection pane taller than necessary, and the log is a diagnostic (scanned, not studied). 2px is an intentional sub-4px exception justified by high-density log readability and limited to this single element. |

**Pre-populated from:** Phase 3 UI-SPEC §Spacing, Phase 4 UI-SPEC §Spacing, Phase 5 RESEARCH Pattern 5 Example CSS.

## Typography

Phase 5 reuses Phase 3 + Phase 4's established typography. Sizes: 3 distinct values (14px, 13px, 12px). Weights: 1 distinct value (regular 400). Line-height: 1.4 (matches Phase 3/4 body).

| Role | Size | Weight | Line Height | Usage in Phase 5 |
|------|------|--------|-------------|------------------|
| Body | 14px | 400 | 1.4 | Connect-button label; Connection pane `<summary>` text |
| Label | 13px | 400 | 1.4 | Connection pane `<legend>` text ("Serial configuration"); serial-config `<label>` text (baud / data bits / etc.); `Reset to MicroBeast preset` button; `Cancel` button next to paste progress; `Paste test` button (inside Debug pane) |
| Hint | 12px | 400 | 1.4 | Port-status line ("Not connected" / "MicroBeast (CP2102N 10c4:ea60)"); paste-progress line ("Pasting 5120 B — 43%"); inline error-log entries (timestamps + messages); Connection pane hint text; polite-fail page body paragraphs |
| Display (polite-fail heading) | 20px | 400 | 1.3 | **One-off heading for the polite-fail full-page takeover only** — `BestialiTTY requires a Chromium-based browser` — single use, never appears in regular chrome. Not a heading system. |

**Rationale for 20px Display (new):** The polite-fail page is a full-page takeover with no surrounding chrome. It needs a single headline that is distinctly larger than body so the user's first glance tells them they're not in the app. 20px is **below the 4-max typography budget** (3 existing + 1 new = 4 total; ≤ 4 sizes satisfied). It exists ONLY on the polite-fail page. Regular app chrome continues to use only 14/13/12.

**Weights budget:** Phase 3 and Phase 4 both committed to a single weight (400 regular). Phase 5 continues that — 1 weight total, satisfies ≤ 2 weights.

**Decision point if 20px Display is rejected by checker:** Collapse to 16px (one step above body). Both are acceptable; 20px is preferred for polite-fail hierarchy.

**Pre-populated from:** Phase 4 UI-SPEC §Typography (14/13/12 scale); Phase 5 adds 20px Display for polite-fail page only (within the ≤ 4 sizes budget).

## Color

Phase 5 adds **two new color hex values** for Connect-button state-border encoding: amber `#e0b030` (connecting/reconnecting) and red `#e04040` (port-lost). These are **state-signal colors**, not palette/accent colors — they never appear outside the `#connect-button[data-state="..."]` selector. All other colors come from the Phase 3 CSS custom-property block at `www/index.html:24-40`.

### Connect-button state-border palette (new — Phase 5)

| State | Border color | Hex / token | Label | Rationale |
|-------|-------------|-------------|-------|-----------|
| disconnected | gray | `var(--chrome-border)` — `rgba(255,255,255,0.08)` | `Connect` | Matches every other top-bar button's default border (Phase 3); "off" state is visually indistinguishable from ambient chrome |
| connecting | amber | `#e0b030` | `Connecting…` | Warm yellow/orange; readable on both `#1e242c` (clean) and `#0a0a0a` (CRT) backgrounds; distinct from phosphor amber (`#ffb000`) which is reserved for terminal glyphs — chrome amber is slightly duller so a user with amber phosphor active doesn't confuse the top-bar signal with terminal content |
| connected | green (phosphor-fg) | `var(--phosphor-fg)` — `#33ff66` (green phosphor) or `#ffb000` / `#e8e8d8` for amber/white | `Disconnect` | Intentionally picks up the active phosphor's foreground color for daily-driver coherence — a green-phosphor user sees a green "connected" signal; an amber user sees amber. **Exception:** on the Connect button specifically (chrome surface), we pin this to `#33ff66` (green phosphor's `fg` value) even when amber/white phosphor is active — rationale: "connected" is a universal success semantic and green is the universal convention. Declaring this on the `[data-state="connected"]` selector directly (with a literal hex, not `var(--phosphor-fg)`) keeps the semantic portable across phosphors. |
| port-lost | red | `#e04040` | `Reconnect` | Muted red (not saturated `#ff0000`) so it reads as "attention" not "panic"; still legible on both theme backgrounds; avoids conflict with any phosphor variant |
| reconnecting | amber | `#e0b030` (same as connecting) | `Reconnecting…` | Same visual state as connecting — they are the same user-facing semantic ("in-progress handshake"); label differentiates, border does not |

**Decision point for connected-border value:** Research Pattern 5 Example CSS proposes `border-color: var(--phosphor-fg)` to inherit the active phosphor. CONTEXT D-01 proposes "green variant regardless of active phosphor". Resolving in favor of CONTEXT D-01 (**green regardless of phosphor**) — universal green success semantic beats phosphor-coherence for a daily-driver chrome button. Implementation: `#connect-button[data-state="connected"] { border-color: #33ff66; }` (literal hex), not `var(--phosphor-fg)`.

The polite-fail page renders BEFORE any theme / phosphor token is applied (before wasm + canvas boot). It MUST NOT use `var(--chrome-*)` custom properties — those get set inside the `<style>` `:root` block, which IS loaded, but the polite-fail page body-swap happens in `main.js` on the very first line, so the safest contract is: **polite-fail uses inline literal colors, not tokens.**

| Role | Value | Usage |
|------|-------|-------|
| Polite-fail background | `#0f1419` | Polite-fail body background — literal, not token; matches clean-theme canvas bg (Phase 3) so even if tokens haven't applied, the visual is consistent |
| Polite-fail foreground | `#e4e8ee` | Polite-fail text — literal, not token; matches clean-theme fg |
| Polite-fail link | `#7fdbca` | Underlined link to Chromium release page — literal, not token; matches clean-theme accent |
| Polite-fail muted | `rgba(255,255,255,0.6)` | Secondary info text |

**Accent reserved for (explicit list):**

1. `:focus-visible` outline on Connect button (existing — on top-bar buttons since Phase 3)
2. `:focus-visible` outline on Connection pane form controls (baud `<select>`, data-bits `<select>`, etc.)
3. `:focus-visible` outline on `Reset to MicroBeast preset`, `Cancel` (paste), `Paste test` buttons (matches Phase 4 `#debug button:focus-visible` rule)
4. Polite-fail link color (literal `#7fdbca`, not the token — but semantically the same accent)

Accent is **NOT** used for: Connect button borders (those use state colors), Connection pane borders (secondary), error-log text (foreground), hint text (muted), port-status text (foreground), paste-progress text (foreground).

### Copywriting invariants (inherited from Phase 4 + tightened for Phase 5)

- **No emojis anywhere.** Pure ASCII / English (Unicode `…` for ellipses is allowed).
- **No exclamation marks.** Calm, informational tone — this is a daily-driver diagnostic.
- **No "please" in UI strings.** We are not asking permission; we are stating state.
- **Byte values always in hex with `0x` prefix when written inline in prose** (matches Phase 4 rule; Phase 5 rarely shows byte-level copy except in hints about flow-control quirks).
- **VID/PID displayed as `XXXX:XXXX` lowercase hex with colon separator** (e.g. `10c4:ea60`) — matches the Linux `lsusb` convention familiar to any retrocomputing user.
- **Serial config notation: `19200 8N1`** (baud + data bits + parity letter + stop bits, space-separated) — canonical telecom shorthand.
- **Event-code names** (inline error log): lowercase kebab-case (`open-failed`, `read-error`). Machine-readable so Playwright assertions can regex on them; user reads the human message after the colon.
- **No "Loading…" spinners anywhere.** State is captured in the Connect button label (`Connecting…`, `Reconnecting…`); there is no separate global loading indicator.
- **No modal dialogs.** Chromium's native `requestPort()` picker is the only non-dismissable chrome in the flow; every other state is an inline UI affordance.

### CSS additions

Phase 5 adds CSS rule blocks that **mirror** existing Phase 3/4 blocks. No new custom properties. Two new literal state-colors are inlined in the `#connect-button[data-state]` selectors only.

```css
/* ==== Connect button in top-bar (Phase 5 D-01, RESEARCH Pattern 5) ==== */
#connect-button {
  padding: 4px 10px;                          /* see Spacing Exceptions */
  background: transparent;
  color: var(--chrome-fg);
  border: 1px solid var(--chrome-border);     /* default gray — state: disconnected */
  cursor: pointer;
  font-family: inherit; font-size: 14px;
  /* No transition on border-color — state changes are discrete events; an
     animation would make connecting-flicker feel like "loading..." UI. */
}
```

## Interaction & State Contracts

These are the load-bearing interaction rules Phase 5 MUST implement. Mostly invisible to the user but critical to phase success.

### Connect button state machine (CONTEXT D-01, D-03, D-04, D-05, D-25)

| Current state | Trigger | Next state | Side effects |
|---------------|---------|------------|--------------|
| `disconnected` | click | `connecting` | Call `navigator.serial.requestPort({ filters: [{ usbVendorId: 0x10c4, usbProductId: 0xea60 }] })`, then `port.open(config)`, then `setSignals({DTR:false, RTS:false})`, then `startReadLoop` |
| `connecting` | `open()` resolves | `connected` | Label → `Disconnect`; border → green `#33ff66`; port-status line updates; persist VID/PID to localStorage |
| `connecting` | `open()` rejects | `disconnected` | Append `open-failed` error-log entry; border returns to gray; label returns to `Connect` |
| `connecting` | user clicks button again | (no-op) | Button is effectively disabled during `connecting` via CSS `pointer-events: none` OR the click handler guards on state; UI feedback is the `…` suffix in the label |
| `connected` | click | `disconnected` | `setSignals({DTR:false, RTS:false})` → `reader.cancel()` → `port.close()`; reader exits via the `done: true` path |
| `connected` | `disconnect` event from `navigator.serial` (USB unplug) | `port-lost` | Append `read-error` or fire via the read-loop's error catch; label → `Reconnect`; border → red |
| `port-lost` | `connect` event from `navigator.serial` (USB re-plug, VID/PID matches, exact ref matches) | `reconnecting` | Auto-open path (CONTEXT D-03, D-25); label → `Reconnecting…`; border → amber |
| `port-lost` | click | `connecting` | User-forced re-authorize: call `requestPort()` with filter again |
| `port-lost` | `connect` event, VID/PID matches but multiple matches AND no exact-ref match | `port-lost` (ambiguous) | Label → `Choose MicroBeast…`; border stays red; append `multiple-adapters` error-log entry |
| `reconnecting` | `open()` resolves | `connected` | Same as `connecting` → `connected` transition |
| `reconnecting` | `open()` rejects | (retry once after 500ms per D-04) | First failure: internal retry silently. Second failure: `port-lost` + `reopen-failed` error |

**UI state display is decoupled from the state machine via `onStateChange(fn)` observer pattern** (RESEARCH Pattern 5 code example). The button updates on observer callback, the Connection pane port-status line updates on the same callback, the error-log is appended by error handlers separately. One source of truth (`state` variable in `serial.js`), many view updates.

### Connect button pointer-events during transient states

`#connect-button[data-state="connecting"]`, `#connect-button[data-state="reconnecting"]` → `pointer-events: none; opacity: 0.7;` — visually dimmed, cursor stays default, click does nothing. Prevents user double-clicking a mid-handshake button. Keyboard activation (Tab + Enter) is also inert because the click handler short-circuits when state is transient.

### Connection pane form-control behaviors

| Control | On change | Persistence |
|---------|-----------|-------------|
| `#serial-baud` `<select>` | Update in-memory config; if connected, **re-negotiation requires disconnect+reconnect** (Web Serial does not support live reconfiguration). Show an inline hint below the select: `Change takes effect on next Connect` — only when `state === 'connected'` and the value differs from the active port's baud. | In-memory only (Phase 6 PREF-01) |
| `#serial-databits`, `#serial-stopbits`, `#serial-parity`, `#serial-flowctl` | Same as baud | In-memory only |
| `#serial-reset-preset` button | Snap all 5 selects to MicroBeast defaults (19200 / 8 / 1 / none / none). If connected, same "takes effect on next Connect" inline hint. | n/a — mutates the in-memory config only |

**Inline reconnect-required hint:** a `<span>` below the fieldset, hidden by default, shown when the active config differs from the open-port config: `Config changed — Disconnect and Connect to apply`. Clears on next Connect.

### Paste-pump UI interactions (CONTEXT D-17, D-18, D-20)

| Trigger | UI reaction |
|---------|-------------|
| `enqueuePaste(bytes)` called | `#paste-progress-row` (in `#top-bar`) becomes visible; text reads `Pasting {N} B — 0%` (Plan 09 amendment — see amended D-17; pane state is irrelevant, no auto-expand) |
| Each chunk written | `#paste-progress-text` updates to new progress value |
| Click `#paste-cancel` | `pastePump.cancelPaste()`; text becomes `Paste cancelled` for 2s then row hides; pane returns to pre-paste open/collapsed state |
| Press Escape while pump active | Same as clicking Cancel (CONTEXT D-18) — `keyboard.js` Esc handler calls `pastePump.cancelPaste()` and does NOT emit 0x1B |
| Paste completes | Text becomes `Paste complete` for 2s then row hides; pane returns to pre-paste state |
| Port lost mid-paste | Text becomes `Paste cancelled — port lost ({N} bytes unsent)` for 3s then row hides |

**Paste progress does NOT use `aria-live`** — it's a high-frequency counter (~55 Hz) that would spam screen readers. It's a visual indicator only. Status transitions (complete / cancelled / port-lost) are not announced either; Phase 5 does not ship screen-reader support for any pane (consistent with Phase 3/4; PROJECT.md Out-of-Scope).

### Focus retention on Phase 5 control click (extends CONTEXT Phase 4 D-16)

All Phase 5 top-bar + Connection-pane + Debug-pane buttons MUST retain `#terminal-wrapper` focus on mousedown, matching the Phase 4 pattern. This preserves the "click a control and keep typing" daily-driver feel.

| Trigger | Element | Behaviour |
|---------|---------|-----------|
| `mousedown` | `#connect-button` | `e.preventDefault()` to retain terminal focus; click handler fires normally and toggles connection |
| `mousedown` | `#serial-reset-preset` | `e.preventDefault()`; click handler fires |
| `mousedown` | `#paste-cancel` | `e.preventDefault()`; click handler fires |
| `mousedown` | `#paste-test` | `e.preventDefault()`; click handler fires |
| `mousedown` | Connection pane `<select>` controls (baud, databits, etc.) | **NO preventDefault** — native `<select>` needs the mousedown to open the dropdown. Accept focus transfer as the cost of using native pickers. User clicks back on terminal to refocus. |
| Keyboard activation (Tab + Space/Enter) | all above | no preventDefault needed; native path works |

### Keyboard shortcuts introduced

Phase 5 introduces **zero new keyboard shortcuts.** Connect/Disconnect is click-only (the `#connect-button` can be Tab-reached and Space-activated via native focus, but there is no chord shortcut). Paste test is click-only. Cancel-paste reuses the existing Escape key behavior (conditional on `pastePump.isActive()`).

**Rationale:** Phase 3 shipped `Ctrl+Alt+T` (theme toggle) and `Ctrl +/-/0` (zoom). Phase 4 shipped no new chords. Phase 5 continues the minimal-chord posture — every new chord is a risk of Chromium conflict (Phase 3 learned this the hard way with `Ctrl+Shift+T`). A click-only Connect is fine because the user only connects once per session; the button is large and persistent.

### Error-log entry lifecycle

1. Any error path appends `{HH:MM:SS} {event-code}: {message}` as a new `.log-entry` `<span>` at the **top** of `#error-log` (newest-first).
2. When the log has > 5 entries, the oldest (bottom) entry is removed from the DOM.
3. Pane auto-expands on new entry (CONTEXT D-27).
4. No UI to clear the log — reload resets; ring-buffer behavior reclaims.

## Accessibility Contract

### Focus order (Tab sequence in Phase 5)

Starting from a freshly loaded page with focus on `#terminal-wrapper`:

1. `#terminal-wrapper` (default focus at boot)
2. `#connect-button` (Phase 5 new — leftmost in top-bar, first tabstop after wrapper)
3. `#theme-toggle` (Phase 3)
4. `#phosphor-group` buttons (Phase 3)
5. `<summary>` of `<details id="connection">` (Phase 5 new)
6. If connection pane expanded: `#serial-baud`, `#serial-databits`, `#serial-stopbits`, `#serial-parity`, `#serial-flowctl` (5 native `<select>`s, Tab advances)
7. If connection pane expanded: `#serial-reset-preset`
8. If paste active: `#paste-cancel`
9. `<summary>` of `<details id="settings">` (Phase 4)
10. (Phase 4 Settings pane tabstops)
11. `<summary>` of `<details id="debug">` (Phase 3)
12. (Phase 3/4 Debug pane tabstops: textarea, Feed, 64 KB Stress, **Paste test (Phase 5 new)**, TX reset)

**Rationale:** Native tab order via DOM order. No `tabindex` beyond what Phase 3 established. No focus traps. Collapsed `<details>` panes skip their internal tabstops automatically (native behavior).

### Semantic markup

- `<button>` elements — not `<div role="button">`. All interactive affordances are native buttons.
- `<select>` + `<option>` — native picker; Chromium renders accessible names from the associated `<label>`.
- `<details>` + `<summary>` — native disclosure; no ARIA needed.
- `<fieldset>` + `<legend>` — native grouping for the serial-config form.
- `<label for="...">` for every form control — native label binding.
- `<a href="...">` on the polite-fail page — real link, not a button with JS navigation.

### Port-status region — live announcement

`#port-status` has `aria-live="polite"` so state changes (connected → port-lost) are announced by screen readers without interrupting. Timestamped error-log entries do NOT get `aria-live` (too noisy; 5 entries on a single-event-storm would spam).

```html
<p id="port-status" class="hint" aria-live="polite">Not connected</p>
```

### Connect button — accessible name

The Connect button's label text IS the accessible name. Tooltips (`title` attribute) provide the verbose state description for sighted users; screen-reader users hear the concise label (`Connect` / `Disconnect` / `Reconnect`). This is WCAG-compliant — accessible name matches visible label.

### Screen reader note (inherited from Phase 4)

PROJECT.md Out-of-Scope lists "Screen-reader support on canvas" as deferred. Phase 5 chrome IS accessible (standard DOM, standard labels, aria-live on port-status). The canvas terminal content is not. Asymmetry is intentional and documented.

<objective>
Fix Gap 2 (paste auto-expands Connection pane and lurches canvas — UAT Test 6, major) AND amend the source-of-truth spec artifacts (05-CONTEXT.md D-17, 05-UI-SPEC.md auto-expand rules) to reflect the new behavior.

**Purpose:**
The paste-progress observer at www/main.js:307-308 explicitly sets `connectionPane.open = true` on the 'started' paste-progress event — faithfully implementing spec D-17 / UI-SPEC §Connection pane auto-expand rules. Because `<details id="connection">` sits above `#terminal-wrapper` in the body's normal flex-column flow, expanding the pane (~250-330 px height delta) translates 1:1 into the canvas's vertical position, which the user reads as "alarming lurch."

This is a SPEC BUG, not a code-vs-spec drift. The implementation matches D-17 verbatim; D-17 is wrong on real hardware. Fix: relocate the paste-progress UI out of `<details id="connection">` and into `#top-bar` (which is `position: sticky; top: 0` and therefore rides at the viewport top without pushing the canvas down). Remove the `connectionPane.open = true` mutation and the associated `preExpansionOpen` capture/restore dance from main.js. Amend D-17 in 05-CONTEXT.md and the UI-SPEC auto-expand rules table in 05-UI-SPEC.md so future gsd-verify / gsd-research runs do not re-propose the old behavior.

Secondary auto-expand at serial.js:410 (D-27 error-log) is acknowledged and kept (errors are rare, sticky, and the red border on the Connect button is the primary signal — auto-opening once per error is acceptable). Documented decision in the D-27 comment amendment.

**Output:**
- Paste-progress row relocated from `<details id="connection">` into `#top-bar` as a `[hidden]`-toggled flex item.
- `connectionPane.open = true` removed from main.js paste observer; preExpansionOpen variable + restore dance removed entirely.
- CSS in www/index.html adds `#paste-progress-row` styling appropriate for the top-bar context (inline flex item, vertical-centered, text-overflow-handled).

# Phase 11 — UI Design Contract

> Visual and interaction contract for Phase 11 (SLIDE JS Bridge & v1.0 Integration). Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Inherits from Phase 3 / 4 / 5 / 6 / 9 / 10.** Phase 3 established the design system (two themes, CSS custom properties, JetBrains Mono + bitmap font, top-bar chrome, `<details>` panes, `[data-focused]` attribute, `atlas.getInverted()`). Phase 4/5/6 added Settings / Connection panes, the `<details class="reserved">` idiom, the stateful Connect button, the inline error log, the prefs-blob pattern, and **the floating `↓ N new lines` chip at `bottom: 8px; right: 8px`** — Phase 11's mirror target. Phase 9 added the top-bar `[↑ Send file]` button + `<dialog>` modal for send confirmation. Phase 10 added the Settings "Save received files to a folder" row, IndexedDB folder-handle persistence, and the Esc-disambiguation slot 2 (SLIDE cancel).
>
> **Phase 11 extends, never redesigns.** It adds **one new floating chip** (`#slide-chip` at `bottom: 8px; left: 8px` — the mirror-image partner to the Phase 6 scrollback chip), **one nested `<details class="reserved">` Settings sub-block** containing four rows (the Phase 10 "Save to folder" row migrated in + three new rows: auto-send command text input, show-summary checkbox, Compatibility mode `<select>`), and **one transient flash state** (drop-rejected) on the chip. Zero new color hex values, zero new typography sizes, zero new spacing tokens, zero new font families, zero new component-library dependencies.
>
> **Hard architectural constraint:** Phase 11 is JS / HTML / CSS only. Zero Rust changes. The Phase 11 chip and Settings rows do not modify the wasm boundary, parser, or terminal state.

### Phase 5 legacy spacing — divergence note (locked)

The existing Phase 5 `#connection select` rule at index.html:371 declares `padding: 2px 4px`. The `2px` value is **not** a multiple of 4 and is **not** in the Phase 11 spacing contract (4, 8, 16, 24, 32, 48, 64). It is a pre-existing legacy carry-forward from Phase 5, **not** a Phase 11 declaration; this UI-SPEC does not retroactively re-style Phase 5's `#connection select`.

**Phase 11 normalizes its own form-control padding to `4px` (a multiple of 4, in the standard set).** The auto-send command `<input type="text">` and Compatibility mode `<select>` introduced by Phase 11 declare `padding: 4px` rather than copying the legacy `padding: 2px 4px` value. This produces a 2px taller form control than the existing `#connection select` (≈ 23px tall vs ≈ 21px tall at `font-size: 13px` + `border: 1px`). The visual difference is minor and the two control sets sit in different Settings panes (Connection vs SLIDE sub-block) so they are not directly adjacent.

If Phase 12 chooses to retroactively normalize the Phase 5 `#connection select` padding to `4px`, that is acceptable polish but explicitly out of scope for Phase 11.

**Pre-populated from:** Phase 6 UI-SPEC §Spacing (chip exception verbatim with `right` → `left`); Phase 10 carry-forward `.settings-row-action { margin-left: 24px }`; CONTEXT C-01.

### Z-index contract (locked — collision policy)

The Phase 6 scrollback chip and Phase 11 SLIDE chip both live at `z-index: 5` per Pitfall 14. They cannot overlap because they sit at opposite corners (`right: 8px` vs `left: 8px`) of the same `position: relative` parent (`#terminal-wrapper`), and each has a max-content width well below half the canvas width (see Layout §Chip max-width below). The bell overlay (`#bell-overlay`) sits at `pointer-events: none; inset: 0` without a declared z-index (relies on DOM order); the scanline overlay similarly. The polite-fail panel sits at `z-index: 6` per index.html:541 — above both chips, by design (a polite-fail dismisses everything below it). **Phase 11 does not change any existing z-index value.**

### Dark theme / canvas-content contrast verification (locked)

The chip background `rgba(0, 0, 0, 0.65)` over arbitrary canvas content (any phosphor color, any glyph, any selection inversion) provides the same contrast on the left side (`#slide-chip`) as it does on the right side (`#scrollback-indicator`) — the canvas content underneath both is statistically equivalent (terminal output character-grid). The Phase 6 chip has been verified to read clearly in both themes; Phase 11's mirror inherits that verification.

## Layout Contract

### Floating chip — DOM shape, lifecycle states, dimensions

**DOM:**
```html
<button id="slide-chip" type="button" hidden
        aria-live="polite" aria-atomic="true"
        aria-label="SLIDE file transfer chip">
  <span id="slide-chip-text"><!-- rendered text per state --></span>
  <!-- Inline buttons rendered as nested <button type="button"> elements
       inside #slide-chip-text on each redraw, e.g.:
       ↑ MY-DOC.TXT  2/3  47%  482 KB  12.3 KB/s  <button class="slide-inline" data-action="cancel">[Cancel]</button>
       The outer #slide-chip itself is a <button> per Phase 6 idiom
       (focusable, click-targetable) but its own click handler is a no-op
       in Phase 11 — only the inner <button class="slide-inline"> targets
       fire actions. -->
</button>
```

**Decision (Claude's Discretion default — `<button>` chip with nested `<button>` buttons):** Phase 6 `#scrollback-indicator` is a `<button>` whose own click snaps to bottom; Phase 11's outer `<button>` has **no click action** (the only actions are on the inner `[Cancel]` / `[Retry]` / `[Force start]` buttons), so the outer `<button>` is functionally a focusable container. This preserves the Phase 6 chip-pattern verbatim and makes the chip Tab-focusable for keyboard users; the inner buttons are then Tab-focusable as second-level targets. **Nested-button accessibility model:** `<button>` inside `<button>` is technically invalid HTML per the Content Model spec; however, (a) Chromium accepts it without error, (b) Phase 6's chip has a similar model (`<button>` with inline `<span aria-hidden="true">↓</span>`) without the nested-button issue, and (c) we ship for Chromium-only per CLAUDE.md. **Mitigation:** if Phase 12 UAT or accessibility audit surfaces issues, planner may flip outer to `<div role="status">` (already declared in CONTEXT Claude's Discretion as the alternative). For Phase 11, **the contract is `<button>` outer + `<button>` inner**.

**Chip text wrap behavior (locked):**

The chip is a single line with `white-space: nowrap`. If content exceeds `max-width`, **the chip overflows to the right — it does not wrap and does not truncate**. Rationale:

- Wrapping a single-line dense layout to two lines breaks the "minimal vertical real estate" intent of D-01.
- Truncating with `text-overflow: ellipsis` would hide tokens (most importantly the throughput and `[Cancel]` button), which defeats the chip's purpose.
- 8.3 CP/M filenames cap at 12 chars, so the active-state worst case is bounded and fits comfortably on a 80-column canvas (which is `80 × char_width_at_18px ≈ 80 × 10.8 ≈ 864px` — the chip's ~460px max sits well inside the left half).
- On very small viewports (< 800px wide canvas), the chip may visually overlap the scrollback chip if both are present simultaneously; this is acceptable because (a) the user is unlikely to be both scrolled-back AND mid-SLIDE-transfer on a tiny window, (b) the SLIDE chip is the higher-priority signal in that case, (c) the overlap is visual only — `pointer-events` for both chips remain on their own bounding box, so click targets don't collide.

**Token separators (locked):** Each separator between chip tokens (between arrow and filename, between filename and N/M, etc.) is **two ASCII spaces** (`  `). The dense layout reads as: `↑·MY-DOC.TXT··2/3··47%··482 KB··12.3 KB/s··[Cancel]` where `·` = ASCII space. This is preserved verbatim from CONTEXT D-01. The two-space separator survives JetBrains Mono's character grid and produces ~12px of visual gap at 12px font-size.

**Throughput unit alignment (locked):** Throughput renders right-aligned in the natural inline flow — Phase 11 does NOT use `display: inline-block` with fixed-width columns. The unit suffix is always present (`B/s`, `KB/s`, `MB/s`, or `B/s` again as the placeholder unit when value `—`); the value preceding it is always present (a digit string OR `—`); the per-character monospace width keeps the trailing `[Cancel]` button position stable across throughput-value updates within the same unit-suffix scale. Cross-suffix transitions (e.g. `999 B/s` → `1.0 KB/s`) shift the right edge of the chip by 1 character cell; this is acceptable.

### Drop-rejected flash — interaction contract

**Trigger:** `slideChip.flashDropRejected()` is called from `file-source.js:178` (onDragEnter when `isSessionActive() && dataTransfer.types.includes('Files')`) and `file-source.js:208` (onDrop with the same predicate).

**State change:** Sets `dropRejectedUntil = Date.now() + 3000`. The next `refreshChip()` invocation (immediate via state-change observer + every 250 ms via tick) detects `Date.now() < dropRejectedUntil` and renders the alternate copy `Transfer in progress — cancel first` IN PLACE OF the active-state content (the chip's outer DOM stays the same; only the inner text content swaps).

**Auto-revert:** On the first `refreshChip()` after `Date.now() >= dropRejectedUntil`, the chip re-renders with the active-state content. No transition / fade / animation — the swap is instant per Spacing exception.

**Edge cases:**
- If the user drops repeatedly during an active session, each call re-extends `dropRejectedUntil` to `Date.now() + 3000` (sliding 3 s window). This is intentional — each rejection should reset the message visibility timer.
- If the SLIDE session completes / cancels while the flash is active, the chip transitions to `cancelled-summary` / `sent-summary` / `received-summary` / `hidden` and the `dropRejectedUntil` flag is implicitly cleared (the state machine moves past `active`).

### Plural rule reference

`{N} {file/files}` resolves as: `1 file` when `N === 1`, `{N} files` otherwise. The `{N}` is rendered as the bare integer with no thousands separator (Phase 11 file counts are bounded — typical sessions are 1-10 files; the rare 100+ file session does not need locale formatting in the chip). Bytes-done `{B}` uses no thousands separator either (e.g. `12345 KB` not `12,345 KB`); the unit suffix prevents ambiguity.

## Accessibility Contract

| Concern | Treatment |
|---------|-----------|

| Reduced motion | No animations / transitions in Phase 11. `prefers-reduced-motion` is automatically honoured by virtue of having nothing to reduce. |
| Screen-reader-only state announcements | `aria-live="polite"` on the chip outer covers state transitions. The drop-rejected flash IS announced (`aria-label` updates → polite live region).

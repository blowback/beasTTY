# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6 (Daily-Driver Polish, Session & Deployment). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Inherits from Phase 3 + Phase 4 + Phase 5.** Phase 3 established the design system (two themes, CSS custom properties, JetBrains Mono + bitmap font, top-bar chrome, Debug `<details>` pane, `[data-focused]` attribute pattern, `atlas.getInverted()`). Phase 4 added the Settings `<details>` pane pattern + the `mousedown preventDefault` focus-retention rule. Phase 5 added the Connection `<details>` pane, the stateful Connect button, the sticky-top-bar paste-progress region, and the inline error log.
>
> **Phase 6 extends, never redesigns.** It adds **seven new chrome surfaces** (one floating canvas overlay + six static buttons/rows) plus **two invisible visual contracts** (`[data-scrolled-back]` border tint + selection rendering via `atlas.getInverted`):
>
> 1. Floating `↓ N new lines` chip — bottom-right of `#terminal-wrapper`, theme-aware, only visible when scrolled up
> 2. Selection rendering on canvas — inverted glyphs via Phase 3 `atlas.getInverted()`, zero new render code
> 3. Top-bar `Clear` button — single new button, slots in next to Connect / theme / phosphor, follows existing top-bar CSS verbatim
> 4. Connection-pane `Download log` button — slots into existing `<details id="connection">` next to `Reset to MicroBeast preset`
> 5. Settings-pane row: `Clear scrollback` button (no shortcut)
> 6. Settings-pane row: `Auto connect on load` checkbox (off by default)
> 7. Settings-pane row: `Reset all preferences` button (inline 2-click confirm)
> 8. Large-paste warn — inline confirm chip in the existing Phase 5 `#paste-progress-row` (`#top-bar` sticky region) when bytes ≥ 4096
> 9. `[data-scrolled-back]` attribute on `#terminal-wrapper` — drives subtle border tint via CSS, mirroring Phase 3's `[data-focused]` pattern
>
> Phase 6 introduces **zero new color hex values**, **zero new spacing tokens**, **zero new typography sizes**, **zero new font-family declarations**, **zero new component-library dependencies**, and **zero new keyboard chord conflicts** (Ctrl+Shift+C / Ctrl+Shift+V are intercepts of existing modifier combos, not new chords in the Chromium sense).

## Color

Phase 6 introduces **zero new color hex values**. Every Phase 6 surface uses tokens already declared in `www/index.html:24-40` (Phase 3) or literal hexes already established in Phase 5 (`#33ff66` / `#e0b030` / `#e04040`).

The two genuinely new visual contracts are:
1. **Floating `↓ N new lines` chip** — uses `var(--phosphor-fg)` (CRT) or `var(--chrome-accent)` (clean) for the text + arrow + border; background is a translucent darkening of `var(--chrome-bg)` (defined inline as `rgba(0,0,0,0.65)` for legibility over any phosphor or canvas content).
2. **`[data-scrolled-back]` border tint on `#terminal-wrapper`** — uses `var(--chrome-accent)` at reduced alpha. Implementation: `border-color: rgba(127, 219, 202, 0.4)` for clean theme; `border-color: var(--phosphor-fg)` at 0.4 alpha for CRT (achieved via `color-mix` if supported, or a literal-per-phosphor table — see CSS Implementation Notes). The intent is "lit but secondary" — strong enough to signal scrolled-up state, weak enough not to compete with `[data-focused]` (which uses full-opacity `var(--chrome-accent)`).

### Selection rendering colors

Per CONTEXT D-20: selection cells are painted by calling `atlas.getInverted(ch, theme, phosphor, …)` — the **same code path** that paints the focused block cursor (Phase 3 D-07). This means:

- **CRT theme, green phosphor:** selected glyphs render as `#0a0f0a` (bg) on `#33ff66` (fg) — i.e. dark glyph on green block.
- **CRT theme, amber phosphor:** `#140d00` on `#ffb000`.
- **CRT theme, white phosphor:** `#0a0a0a` on `#e8e8d8`.
- **Clean theme:** `#0f1419` (bg) on `#e4e8ee` (fg) — dark glyph on light block.

No new color tokens are introduced — selection inversion is **pure reuse** of existing per-theme `{fg, bg}` pairs via the established Phase 3 atlas mechanism.

### Floating chip color contract (the only genuinely new color usage)

The floating `↓ N new lines` chip is the single Phase 6 surface that introduces a new color *combination* (not a new color value). It uses existing tokens but in a configuration not seen elsewhere in the project.

| Element | Color | Theme | Rationale |
|---------|-------|-------|-----------|
| Chip text + arrow | `var(--phosphor-fg)` | CRT | Daily-driver coherence — "new output is happening at the live tail" reads in the same color as the live output itself. Across all three phosphors (green/amber/white), the chip color tracks. |
| Chip text + arrow | `var(--chrome-accent)` (`#7fdbca`) | clean | Clean-theme accent is the closest analog to "phosphor-fg" — both signal "active interactive UI". |
| Chip border | same as text (1px solid) | both | Visual unity — the chip reads as a single token glowing at the canvas edge. |
| Chip background | `rgba(0,0,0,0.65)` | both | Theme-independent dark backdrop — guarantees legibility over ANY canvas content (even a CRT amber-phosphor screen full of text). Literal value, not a token, because: (a) `var(--chrome-bg)` would render the chip nearly invisible on the CRT theme (chrome bg matches canvas bg); (b) no other Phase 6 surface uses this backdrop, so introducing a token would inflate the design system for one use. |
| Chip drop-shadow | `0 2px 8px rgba(0, 0, 0, 0.5)` | both | Subtle elevation — the chip floats over canvas content; a shadow signals "z-index above terminal grid". Theme-independent dark shadow because the chip sits on canvas-bg-colored content. |

## Copywriting Contract

All strings below are locked by 06-CONTEXT.md decisions (D-03, D-15, D-25, D-26, D-31, D-34, D-35, D-36) plus researcher-composed microcopy for the gaps CONTEXT left to discretion. Planner and executor MUST use these exact strings — no paraphrasing.

### Floating `↓ N new lines` chip (CONTEXT D-03)

| Element | Copy (verbatim) | Notes |
|---------|-----------------|-------|
| Chip text (1 new line) | `↓ 1 new line` | Singular — 1 line ≠ "lines" |
| Chip text (N > 1) | `↓ {N} new lines` | Plural — `{N}` formatted with locale-default thousands separator (`1,234`) for 4+ digit values; raw integer for 1–999 |
| Chip text (N ≥ 1000, optional) | `↓ {N} new lines` | Same as above; thousands separator handles large values. NO truncation to `999+` — daily-driver users want to know "is this a 5K-line dump or a 50K-line dump"; locale-formatted thousands separators are universally legible. |
| Chip tooltip (`title` attribute) | `Click to scroll to live output` | Single static string — does not change with N |
| Chip `aria-label` | `{N} new lines below — click to scroll to live output` | Screen reader gets the count + the action; sighted user gets the visual count + the tooltip on hover. |

**Arrow character:** Unicode `↓` (U+2193, "DOWNWARDS ARROW"). Not `▼` (filled triangle, too heavy), not `v` (ASCII letter, ambiguous), not `\u{1F53D}` (emoji 🔽, violates no-emoji rule).

**Plural rule:** English plurals only — `1 new line` / `2 new lines` / `0 new lines`. The chip only renders when `N >= 1` so the zero-form never appears in practice; including it for completeness.

### Large-paste inline confirm chip (CONTEXT D-25)

Lives inside the existing Phase 5 `#paste-progress-row` (inside `#top-bar`). Renders only when paste size ≥ 4096 bytes; smaller pastes start immediately (Phase 5 progress UI is sufficient).

| Element | DOM ID / Role | Copy (verbatim) |
|---------|---------------|-----------------|
| Confirm prompt text | text content of `#paste-progress-text` while in confirm mode | `About to paste {N} B (~{S} s at {BAUD} baud).` (e.g. `About to paste 100,234 B (~52 s at 19200 baud).`) |
| Cancel button | `<button id="paste-cancel">` | `Cancel` |
| Paste (commit) button | `<button id="paste-confirm">` (NEW — only present in confirm mode) | `Paste` |

**Format details:**

- `{N}`: decimal byte count with locale-default thousands separator (`100,234`, not `100234`).
- `{S}`: estimated seconds, computed as `ceil(N * 10 / baud)` (10 bits per byte at 8N1 — start + 8 data + stop). Round to integer; show `~{S} s` (lowercase `s` for seconds; `~` indicates approximation).
- `{BAUD}`: integer baud value from current serial config (default `19200`).

**Behavior:** Pump does NOT start until user clicks `Paste`. Cancel clears the pending bytes (no clipboard re-read needed; nothing has been written to the wire yet). Below the 4096-byte threshold, paste starts immediately and the existing Phase 5 progress UI takes over.

**Threshold rationale (informative — not user-facing copy):** 4096 bytes ≈ 2.1 seconds at 19200 baud — long enough that an unintended paste produces visible terminal disruption, short enough that genuine quick pastes (a config one-liner, a short command) don't trigger the confirm step.

### `[data-scrolled-back]` border tint — no copy

Pure visual contract. No tooltip, no label. The border tint is the entire signal; the floating chip provides the actionable copy.

### Selection — no UI copy

Selection is a pure visual/clipboard interaction. No tooltip, no banner, no toast. Successful copy clears the selection (D-19) — the lack of a visible selection IS the success signal. Empty-selection + Ctrl+Shift+C is a silent no-op (D-21) — no user-facing feedback, because the user clearly didn't have a selection to copy.

### `<title>` — Phase 6 inheritance

Phase 6 does NOT modify the document title. Phase 3's `(!) BestialiTTY` BEL prefix and Phase 5's `BestialiTTY — Chromium required` polite-fail title both remain unchanged. The Phase 6 floating chip is the dedicated "scrolled-up new output" signal — not the title.

### Copywriting invariants (inherited from Phase 4 + Phase 5, tightened for Phase 6)

- **No emojis anywhere.** Pure ASCII / English (Unicode `…` and `↓` are allowed — `↓` is the floating chip arrow per D-03).
- **No exclamation marks.** Calm, informational tone — this is a daily-driver UI.
- **No "please" in UI strings.** We are not asking permission; we are stating state or action.
- **No "Are you sure?" prompts.** All confirmations are inline 2-click chips (D-25 large-paste, D-35 reset-prefs). No modal dialog infrastructure exists or will exist in v1.
- **Byte values always in hex with `0x` prefix when written inline in prose** (matches Phase 4/5 rule). `{N} B` for decimal byte counts in user-facing labels (large-paste warn, log size).
- **Time values:** seconds use lowercase `s` (`~52 s`); milliseconds use `ms`. No mixed `secs` / `seconds` / `Sec`.
- **Numbers ≥ 1000 use locale-default thousands separator** (`100,234 B`, not `100234 B` and not `100.234 B`).
- **Filename timestamps:** `YYYYMMDD-HHMMSS` (no separators inside date or time, hyphen between date and time). Matches D-31 + Claude's Discretion ruling — chosen for shell-friendly filenames (no colons, no spaces, sortable lexically).
- **Singular vs plural enforced:** `1 new line` / `N new lines` (D-03 chip). Never `1 new lines` / `1 new line(s)`.
- **Tooltips end without punctuation** (no trailing period). Body-text hint paragraphs end with a period.
- **No "Loading…" spinners anywhere.** Phase 6 has no async UI states that need spinner indication; localStorage reads/writes are synchronous, log downloads are instant from the user's perspective, snap-to-bottom is instant.
- **No modal dialogs.** Inherited from Phase 5. Inline confirms (D-25, D-35) are the only confirmation pattern. Native browser confirms (`window.confirm`) are also forbidden — they steal focus and break the daily-driver flow.

**Justification for top-bar Clear button placement (between Connect and theme-toggle):** Connect carries the highest-priority phase signal (border color = connection state); leftmost is correct. Clear is the second-most-frequent action in daily-driver use (clearing the screen is a several-times-per-session action; theme toggle is a once-per-session-or-less action). Therefore Clear sits second-leftmost. Theme toggle and phosphor radio-group remain on the right (rarely used). This ordering optimizes for left-to-right scan: state → frequent action → rarely-used controls.

**Justification for Settings-pane DOM order (Clear scrollback → Auto connect on load → Reset all preferences):** Per CONTEXT §Claude's Discretion, planner picks based on Phase 4 D-14. Researcher decision: order by **action scope, narrow → wide** — `Clear scrollback` is a narrow action (only affects scrollback contents), `Auto connect on load` is a behavioral toggle (affects future page loads), `Reset all preferences` is the broadest action (affects everything). Reset is at the bottom because (a) it's the most consequential, (b) putting it last means a user scrolling through Settings sees the safer actions first, (c) the visual settings-divider `<hr>` separating the new block from the existing Phase 4 rows reinforces the "danger increases as you scroll down" gradient.

**Justification for `#scrollback-indicator` z-index = 5:** Bell overlay (Phase 3) has no z-index. Scanlines overlay has no z-index. The chip needs to render ABOVE both. z-index 5 is comfortably above unscrolled siblings without competing with the body-level polite-fail page (which replaces `<body>` content entirely — no z-index conflict possible).

## Interaction & State Contracts

These are the load-bearing interaction rules Phase 6 MUST implement. Mostly invisible to the user but critical to phase success.

### Scrollback navigation state machine (CONTEXT D-01..D-15)

| Current state | Trigger | Next state | Side effects |
|---------------|---------|------------|--------------|
| `live` (`scrollOffset === 0`) | wheel up over `#terminal-wrapper`, accumulator crosses threshold | `scrolled-back` (`scrollOffset > 0`) | Set `[data-scrolled-back="true"]` on `#terminal-wrapper`; pause cursor blink (D-09); skip dirty-row pipeline (D-08); paint all 24 rows once via `snapshot_grid_at(scrollOffset)` |
| `live` | Shift+PgUp keydown | `scrolled-back` (jumped 24 rows) | Same as above; `scrollOffset += 24` |
| `live` | Shift+Home keydown (D-05) | `scrolled-back` at top (`scrollOffset = scrollback_len`) | Same as above; `scrollOffset` clamped to `scrollback_len` |
| `scrolled-back` | new `term.feed()` chunk arrives with line breaks | `scrolled-back` (chip counter increments) | Increment `newLinesSinceUserScrolled`; update chip text + aria-label; render chip if hidden; viewport content does NOT change |
| `scrolled-back` | wheel down past live boundary | `live` | Remove `[data-scrolled-back]`; resume cursor blink; reset `newLinesSinceUserScrolled = 0`; hide chip; resume dirty-row pipeline |
| `scrolled-back` | Shift+End keydown | `live` | Same as wheel-down-past-live |
| `scrolled-back` | click `#scrollback-indicator` chip | `live` | Same — chip click is a snap-to-bottom trigger (D-04) |
| `scrolled-back` | any TX-producing keypress (printable, Enter, arrow, etc.) | `live` | Same — typing is a snap-to-bottom trigger (D-04) |
| `scrolled-back` | `enqueuePaste(bytes)` called | `live` | Same — paste is a snap-to-bottom trigger (D-04) |
| `scrolled-back` | MicroBeast reconnect after port-lost (Phase 5 D-03) | `live` | Same — reconnect is a snap-to-bottom trigger (D-04) |
| `scrolled-back` | `term.resize_scrollback(0)` from Settings 'Clear scrollback' (D-15) | `live` | Same — clear-scrollback is a snap-to-bottom trigger (D-04) |
| `scrolled-back` | theme/phosphor/zoom toggle | `scrolled-back` (offset preserved, repaint) | `markAllRowsDirty()` (Phase 3 pattern) → `snapshot_grid_at(scrollOffset)` repaint with new style; viewport stays at the same row offset (D-13) |
| `scrolled-back` | BEL byte received | `scrolled-back` (no flash) | Skip viewport flash (D-10); apply title prefix `(!) ` only when document.hidden (Phase 3 D-09 unchanged) |
| `live` | wheel down over `#terminal-wrapper` (already at live tail) | `live` | No-op (D-18 mid-line — drag-past-bottom-while-at-live-tail is a no-op; same applies to wheel) |
| any | wheel over `<details id="settings">` / `<details id="connection">` / `<details id="debug">` | (no-op on scrollback) | Native pane scrolling — wheel listener is on `#terminal-wrapper`, not document (D-12) |

**Wheel sensitivity (D-02):**
- `e.deltaMode === DOM_DELTA_LINE`: 3 lines per `deltaY` unit; e.g. `deltaY = -1` (one notch up) → `scrollOffset += 3`. Shift+wheel: 1 page (24 lines) per unit.
- `e.deltaMode === DOM_DELTA_PIXEL` (trackpad): accumulate raw `deltaY`; emit a 3-line tick when `Math.abs(accumulator) >= 30`; reset accumulator on each emitted tick. Threshold `30 px` per Claude's Discretion (CONTEXT §Claude's Discretion); planner verifies on real trackpad.
- `e.deltaMode === DOM_DELTA_PAGE`: rare; treat as 24 lines per unit (page).

**Cursor while scrolled up (D-09):** `paintCursor()` short-circuits when `scrollState.isScrolledBack === true`. The Phase 3 wall-clock blink-rate counter continues to increment internally (no allocation cost); just no paint. Re-enables on snap-to-bottom; first paint after snap shows the cursor in its current blink phase (no jank).

### Selection state machine (CONTEXT D-16..D-21)

| Current state | Trigger | Next state | Side effects |
|---------------|---------|------------|--------------|
| `idle` | pointerdown on `<canvas id="terminal">` (single click) | `dragging` | Capture anchor cell (px → row/col); start tracking pointer; set `pointer-events: auto` on canvas (already default); call `setPointerCapture` for cross-window drag tolerance |
| `dragging` | pointermove | `dragging` (extend) | Update focus cell; mark selection rectangle dirty for repaint; if pointer is past top edge → start auto-scroll up at ~3 lines/sec (D-18); if past bottom while scrolled-up → auto-scroll down |
| `dragging` | pointerup | `selected` | Commit selection range; selection cells render via `atlas.getInverted()` (D-20); fire `onSelectionChange(true)` for any subscribers |
| `dragging` | Esc keydown | `idle` | Cancel in-flight drag; clear selection; no clipboard write (D-19) |
| `selected` | double-click on canvas | `selected` (whitespace-bounded word) | Replace anchor/focus with whitespace-bounded run; repaint selection cells; D-16 |
| `selected` | triple-click on canvas | `selected` (entire row) | Replace selection with full row; repaint; D-16 |
| `selected` | Ctrl+Shift+C keydown | `idle` | `clipboard.writeText(decodedText)`; on success: clear selection (D-19); on failure: keep selection, log error (no UI surface) |
| `selected` | any scroll-state change (wheel, Shift+PgUp/PgDn/Home/End, snap-to-bottom) | `idle` | Clear selection (D-19); selection does NOT survive scroll |
| `selected` | theme/phosphor/zoom toggle | `idle` | Clear selection (D-19); selection does NOT survive style change |
| `selected` | focus loss on `#terminal-wrapper` | `idle` | Clear selection (D-19) |
| `selected` | empty selection + Ctrl+Shift+C | `idle` (no-op) | Silent no-op — no clipboard write, no error UI (D-21) |
| `dragging` | pointerup with anchor === focus (zero-length selection) | `idle` | Treat as click-to-position (no selection commit); no copy |

**Selection endpoint storage (D-17):** endpoints stored as `(scrollback_row_offset_from_live_tail, col)` tuples — stable identifiers across viewport scroll during in-flight drag. Translation to viewport row at paint time: `viewport_row = scrollback_row_offset_from_live_tail - scrollOffset`.

**Drag-past-edge auto-scroll (D-18):** while pointer Y is above canvas top during `dragging`, `setInterval` (or `setTimeout` chain) at 333 ms interval (3 lines/sec) increments `scrollOffset` and re-extends selection. Stops when pointer returns inside canvas OR pointerup. Drag past bottom while at live tail = no-op (no further scroll possible).

**Word boundary regex (D-16, Claude's Discretion):** `/\S+/` — whitespace-bounded run of non-space chars. Planner verifies on real CP/M / BASIC outputs at implementation time. Final regex committed in implementation.

### Clipboard contract (CONTEXT D-21..D-25)

| Operation | Trigger | Side effects |
|-----------|---------|--------------|
| Copy | Ctrl+Shift+C with non-empty selection | `keyboard.js` intercepts; calls `clipboard.copySelection()`; selection text decoded per D-23 (trailing whitespace trimmed per line, `\n` line endings, no trailing `\n` for single-line); `navigator.clipboard.writeText(text)`; on resolve: clear selection, return; on reject: keep selection, console.warn |
| Plain Ctrl+C | (no selection) Ctrl+C | Phase 4 path — encodes 0x03 (ETX), sends to remote. No clipboard interaction. Sacred from D-21. |
| Plain Ctrl+C | (with selection) Ctrl+C | **Same as above** — D-21 locks Ctrl+C as 0x03 unconditionally. Selection state irrelevant; user uses Ctrl+Shift+C to copy. |
| Paste | Ctrl+Shift+V | `keyboard.js` intercepts; calls `clipboard.pasteFromClipboard()` async; on resolve: bytes preprocessed per D-24 (strip non-printable except CR/LF/Tab; apply `crlfMode` rewrite); if `bytes.length >= 4096`: render large-paste confirm chip (D-25); else: `pastePump.enqueuePaste(bytes)` immediately |
| Plain Ctrl+V | Ctrl+V (no shift) | Phase 4 path — encodes 0x16 (SYN), sends to remote. No clipboard interaction. Sacred from D-22. |
| Large-paste confirm — Cancel | click `#paste-cancel` in confirm chip | Discard pending bytes; hide `#paste-progress-row`; `#paste-confirm` button hidden again |
| Large-paste confirm — Paste | click `#paste-confirm` in confirm chip | `pastePump.enqueuePaste(bytes)`; transition `#paste-progress-row` to standard Phase 5 progress UI (`Pasting {N} B — {P}%`); `#paste-confirm` button hidden |

**`navigator.clipboard.readText()` user-gesture requirement:** `Ctrl+Shift+V` keydown IS a user gesture; the call resolves without permission prompt assuming the user has previously granted clipboard-read to the origin OR the Permissions-Policy `clipboard-read=(self)` is in force (deferred to D-39 deployment config). On Permissions-Policy block: `console.warn` + log to inline error log (Phase 5 surface) with code `clipboard-blocked`.

### Session log lifecycle (CONTEXT D-27..D-31)

| Event | Action |
|-------|--------|
| First post-Connect `term.feed(value)` (RX bytes from MicroBeast) | Allocate fresh `chunks: Uint8Array[] = []`; record `connectTimestamp = new Date()`; push the chunk; set `currentLogBytes = chunk.length`; enable `#download-log-button` (remove `disabled`) |
| Subsequent `term.feed(value)` | Push chunk to array; `currentLogBytes += chunk.length`; button stays enabled |
| Click `#download-log-button` (mid-session) | Construct `new Blob(chunks, { type: 'application/octet-stream' })`; create `<a download="bestialitty-{YYYYMMDD-HHMMSS}.bin" href="{blobURL}">`; programmatic click; revoke blob URL after 60 s; chunks array NOT cleared (logging continues) |
| Disconnect (user-initiated or port-lost) | Chunks array preserved through to next Connect; user has one final opportunity to download before next Connect discards it |
| Subsequent Connect | Discard previous chunks array (D-29); allocate fresh; `currentLogBytes = 0`; button returns to disabled state with tooltip `No bytes received yet` |
| `beforeunload` | No-op for log — log is in-memory only; reload discards |

**Filename timestamp format:** `YYYYMMDD-HHMMSS` from connect time. UTC vs local: planner picks based on consistency with existing project conventions; researcher recommends LOCAL time (matches `new Date().toTimeString()` Phase 5 error-log convention) for daily-driver shell-friendliness — user's filesystem timestamps will match wall clock.

### Preferences lifecycle (CONTEXT D-32..D-36)

| Event | Action |
|-------|--------|
| Page load — `loadPrefs()` | Read `localStorage.getItem('bestialitty.prefs')`; if null → use defaults (D-36); else parse JSON; if `version < CURRENT_VERSION` → run migration handler; cache in-memory `prefs` object |
| Any preference change (theme, phosphor, zoom, baud, dataBits, stopBits, parity, flowControl, localEcho, crlfMode, autoConnect) | `savePrefs({ key: value })` merges into in-memory blob; schedules a single `setItem` via `setTimeout(persist, 250)`; subsequent changes within 250 ms reset the timer (debounce) |
| `beforeunload` | Flush pending debounced write IMMEDIATELY (synchronous `setItem` in handler); no `await` (handler must return synchronously per browser contract) |
| `localStorage.setItem` throws QuotaExceededError | Catch; `console.warn`; in-memory prefs preserved for session; no UI surface (no actionable recovery) |
| Click `#reset-prefs-button` (idle state) | Swap label to `Click again to confirm (3 s)`; start 3 s timer; arm second-click handler |
| Click `#reset-prefs-button` (confirm state, within 3 s) | Cancel timer; `localStorage.removeItem('bestialitty.prefs')`; in-memory prefs replaced with defaults; subscribers notified via `prefs.subscribe` callback chain (chrome.js, canvas.js, keyboard.js apply new values); button returns to idle label; **NO page reload** — D-35 |
| Click outside button OR 3 s timeout in confirm state | Restore button label to `Reset all preferences`; cancel armed handler |
| Auto-connect toggle | Save to `prefs.autoConnect`; takes effect on next page load (no immediate connect/disconnect — no auto-connect/disconnect cycle on toggle) |
| Page load with `prefs.autoConnect === true` AND `getPorts()` finds matching VID/PID | After wasm + canvas boot, silently call `connectMicroBeast()` (no user click). On success: standard connected flow. On failure: append `auto-connect-failed` to inline error log (Phase 5 surface); user sees button in `disconnected` state with the failure logged |

### Snap-to-bottom triggers (CONTEXT D-04 — exhaustive list)

ANY of the following transitions `scrolled-back` → `live` and resets `newLinesSinceUserScrolled = 0`, hides the floating chip, removes `[data-scrolled-back]`:

1. Click `#scrollback-indicator` chip
2. Press Shift+End
3. Wheel down past the live tail boundary
4. Any TX-producing keypress (printable char, Enter, arrow keys, function keys, Ctrl-letter, etc. — anything `keyboard.js` would normally send to the remote)
5. `enqueuePaste(bytes)` call (clipboard paste OR Paste test button OR large-paste confirm)
6. MicroBeast reconnect after port-lost (Phase 5 D-03 `connect` event with VID/PID match)
7. Click Settings `Clear scrollback` button → calls `term.resize_scrollback(0)` → triggers snap (D-15)
8. Click Top-bar `Clear` button (plain or Shift+click) — both wipe state and snap
9. Theme/phosphor/zoom toggle does NOT snap (D-13 — viewport offset preserved)

### `[data-scrolled-back]` attribute lifecycle (CONTEXT D-13)

| Event | Action |
|-------|--------|
| `scrollState.isScrolledBack` transitions `false → true` | `terminalWrapper.setAttribute('data-scrolled-back', 'true')` |
| `scrollState.isScrolledBack` transitions `true → false` | `terminalWrapper.removeAttribute('data-scrolled-back')` (or set to `'false'` — CSS selector is `[data-scrolled-back="true"]` so absence is the false state) |
| Initial page load | Attribute absent (`isScrolledBack === false` at boot) |

Mirrors Phase 3's `[data-focused]` attribute pattern verbatim. CSS rule applies a 0.4-alpha accent border (no reflow — same 1px width as `[data-focused]` and as the default transparent border).

### Floating chip lifecycle (CONTEXT D-03)

| Event | Action |
|-------|--------|
| First new line received while `isScrolledBack === true` | Increment `newLinesSinceUserScrolled` to 1; update chip text via `aria-label` and visible `<span>`; remove `hidden` attribute on `#scrollback-indicator` (becomes visible instantly — no transition) |
| Subsequent new lines while `isScrolledBack === true` | Increment counter; update text/aria-label |
| `isScrolledBack` transitions to `false` (snap-to-bottom) | Reset counter to 0; add `hidden` attribute on `#scrollback-indicator` (becomes invisible instantly) |
| Click `#scrollback-indicator` chip | Behaves as snap-to-bottom (D-04 trigger 1); chip hides as side effect |
| `isScrolledBack === true` but no new lines yet | Chip remains hidden (counter is 0) — chip only renders when there is something new to announce |

**Chip update cadence:** chip text updates synchronously after every `term.feed()` that includes `\n` bytes — the same synchronous block that runs `sampleBell()` + `drainHostReply()` + `requestFrame()` (Phase 3 invariant). No rAF delay; no debounce. The chip is a high-frequency counter but it's text-only; DOM updates are cheap.

### Keyboard shortcuts introduced

Phase 6 introduces **two new keyboard intercepts** (not new chords — modifier-extension intercepts of existing combinations):

| Chord | Intercept rule | Action | Sacred — what stays on existing path |
|-------|---------------|--------|--------------------------------------|
| `Ctrl+Shift+C` | `e.ctrlKey && e.shiftKey && !e.altKey && !e.metaKey && e.code === 'KeyC'` | `clipboard.copySelection()`; if no selection, no-op | `Ctrl+C` (no shift) → Phase 4 path → encodes 0x03 (ETX) sacred |
| `Ctrl+Shift+V` | `e.ctrlKey && e.shiftKey && !e.altKey && !e.metaKey && e.code === 'KeyV'` | `clipboard.pasteFromClipboard()` async | `Ctrl+V` (no shift) → Phase 4 path → encodes 0x16 (SYN) sacred |
| `Shift+PgUp` | `e.shiftKey && !e.ctrlKey && !e.altKey && e.code === 'PageUp'` | `scrollState.scrollUp(24)` | `PgUp` (no shift) → Phase 4 path → emits to remote |
| `Shift+PgDn` | `e.shiftKey && !e.ctrlKey && !e.altKey && e.code === 'PageDown'` | `scrollState.scrollDown(24)` | `PgDn` (no shift) → Phase 4 path → emits to remote |
| `Shift+Home` | `e.shiftKey && !e.ctrlKey && !e.altKey && e.code === 'Home'` | `scrollState.jumpToTop()` (D-05) | `Home` (no shift) → Phase 4 path → emits to remote |
| `Shift+End` | `e.shiftKey && !e.ctrlKey && !e.altKey && e.code === 'End'` | `scrollState.snapToBottom()` (D-04) | `End` (no shift) → Phase 4 path → emits to remote |

**`preventDefault()` discipline:** All six chords call `e.preventDefault()` SYNCHRONOUSLY before any async work, per Phase 3 RESEARCH Pitfall #3. `e.code` not `e.key` per Pitfall #10.

**Browser-conflict check:** `Ctrl+Shift+C` is Chromium's "open DevTools inspector" chord on some platforms. **Live testing during Phase 6 implementation is REQUIRED** to confirm `e.preventDefault()` is honored. Phase 3 learned this with `Ctrl+Shift+T` (Chromium-reserved, unpreventable). If `Ctrl+Shift+C` proves unpreventable, fallback chord is `Ctrl+Insert` (Linux-style copy) — recorded as "follow-up if Chromium reserves Ctrl+Shift+C". Researcher position: most modern terminals (gnome-terminal, kitty, Windows Terminal, iTerm2) successfully bind Ctrl+Shift+C; the Chromium DevTools binding is conditional on devtools being open or the Inspect-via-Element flow. Implementation testing will confirm.

### Esc key disambiguation (extends Phase 5 D-18)

The Esc key has THREE possible behaviors at runtime, in priority order:

1. **If `selection.isInFlight`** (`dragging` state): cancel selection, return — D-19. Esc does NOT propagate to remote.
2. **Else if `pastePump.isActive()`** (Phase 5 D-18): cancel paste, return — D-18. Esc does NOT propagate to remote.
3. **Else (default)**: Phase 4 path — encode 0x1B (ESC), send to remote.

This priority order ensures that an in-flight drag + active paste pump + remote-bound Esc never collide. Selection cancel is highest priority because it's a UI-only state with no remote side effect; paste cancel is next; remote ESC is the fallback.

### Quota-error handling

`localStorage.setItem('bestialitty.prefs', json)` on `QuotaExceededError`:

```js
try { localStorage.setItem('bestialitty.prefs', json); }
catch (e) {
  if (e.name === 'QuotaExceededError') {
    console.warn('[prefs] Could not persist preferences (storage quota). In-memory only.');
  } else {
    console.warn('[prefs] Could not persist preferences:', e);
  }
}
```

No UI surface — there is no actionable user recovery (clearing storage is platform-level and the user is already blocked by their own quota). Session-local prefs continue to function.

### Focus order (Tab sequence, Phase 6 — full sequence)

### Semantic markup

- `<button>` for `#clear-button`, `#download-log-button`, `#clear-scrollback-button`, `#reset-prefs-button`, `#paste-confirm`, `#scrollback-indicator` — all native.
- `<input type="checkbox">` for `#auto-connect-checkbox` — native.
- `<a href="..." download="...">` (synthesized at runtime) for log download — real link, programmatic click.
- `<hr class="settings-divider">` for the visual divider between Phase 4 Settings rows and Phase 6 new rows — native semantic separator.

### `aria-live` and announcements

- **`#port-status`** (Phase 5) — already `aria-live="polite"`; unchanged.
- **`#scrollback-indicator`** — `aria-label` updates with new count; the button itself is NOT `aria-live` (announcing every new line on a busy MicroBeast session would spam screen readers). Sighted users see the visual update; non-sighted users hear the count when they Tab onto the chip. This is an intentional asymmetry.
- **Floating chip text content** — uses `aria-hidden="true"` on the visual `↓` arrow so screen readers don't read "DOWNWARDS ARROW"; the `aria-label` on the button covers the full announcement.
- **`#paste-progress-text`** (Phase 5) — already explicitly NOT `aria-live` per Phase 5 spec; unchanged.
- **`#error-log`** (Phase 5) — already explicitly NOT `aria-live`; unchanged.
- **Settings checkbox `#auto-connect-checkbox`** — native `<input>` semantics; checked state announced by screen readers natively.
- **Reset prefs confirm-state label change** — the button text changes from `Reset all preferences` to `Click again to confirm (3 s)`. Screen readers will re-read the changed accessible name on next focus event. This is acceptable; the user is intentionally interacting with the button.

### Selection accessibility

Selection rendering is canvas-only — by definition NOT screen-reader accessible. PROJECT.md Out-of-Scope lists "Screen-reader support on canvas" as deferred; Phase 6 does not introduce screen-reader-accessible selection. Sighted users see the inverted glyphs; non-sighted users access content via the existing canvas content (which is itself not screen-reader accessible — same documented limitation as Phase 3/4/5).

This asymmetry is intentional and inherited. Documented in this UI-SPEC for completeness.

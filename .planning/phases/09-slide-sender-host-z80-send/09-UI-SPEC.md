# Phase 9 — UI Design Contract

> Visual and interaction contract for the SLIDE Sender (Host -> Z80) phase.
> Generated by gsd-ui-researcher; verified by gsd-ui-checker.
>
> **Inherits from Phases 3, 4, 5, 6, 8.** Phase 3 established the design
> system (CSS custom properties, two themes, top-bar chrome, focus-attribute
> idiom). Phase 5 added the Connect-button stateful pattern + Connection
> pane. Phase 6 added the floating-chip pattern, additional top-bar buttons
> (`Clear`), and the `[data-scrolled-back]` attribute idiom on
> `#terminal-wrapper`. Phase 8 added the `Slide` wasm dispatcher (no UI
> surface). **Phase 9 introduces three brand-new chrome surfaces and one
> brand-new HTML idiom**:
>
> 1. A top-bar text button `[↑ Send file]` (CONTEXT D-01) — slots in next
>    to `[Connect] [Clear] [Clean] [Green|Amber|White]`, follows existing
>    `#top-bar button` CSS verbatim. **Disabled-with-suffix-label** while a
>    SLIDE session is pending or active (`[↑ Send file (sending…)]`,
>    CONTEXT D-15 + D-18).
> 2. A hidden `<input type="file" multiple>` adjacent to the button (D-01).
> 3. A drag-drop overlay div parented to `#terminal-wrapper` toggled via a
>    new `[data-drop-target]` attribute (CONTEXT D-03), mirroring the
>    Phase 3 `[data-focused]` and Phase 6 `[data-scrolled-back]` idiom.
> 4. A native `<dialog>` element for the rewrite/rejection confirm modal
>    (CONTEXT D-05), styled to mirror the Phase 6 chip palette.
>
> Phase 9 introduces **zero new typography sizes, zero new spacing tokens,
> zero new font-family declarations, zero new color tokens, zero new
> keyboard shortcuts**. The auto-typed `B:SLIDE R\r` echoes visibly in the
> terminal — Phase 11 owns the swallow-echo filter (SLIDE-14).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla ES modules — PROJECT.md + Phase 2 D-14 + Phase 3..6 UI-SPECs lock framework-free posture) |
| Preset | not applicable |
| Component library | none (native `<button>` / `<input type="file">` / `<dialog>` / `<ul>` / `<li>`) |
| Icon library | none — Unicode `↑` (U+2191 UPWARDS ARROW) is the only "icon" used in chrome (CONTEXT D-01); no SVG, no emoji, no icon font. Renders in the chrome font stack (`ui-monospace, SFMono-Regular, Menlo, Consolas, monospace`) which has full BMP coverage on every supported platform |
| Font (chrome) | JetBrains Mono Regular self-hosted WOFF2 + fallback chain `ui-monospace, SFMono-Regular, Menlo, Consolas, monospace` — inherited verbatim from Phase 3 D-02. The `↑` glyph (U+2191) is part of the standard fallback chain coverage; if JetBrains Mono fails to load (CONTEXT D-13 echo edge case is unrelated) the system fallback renders it correctly. **The arrow lives in the chrome layer, not the in-canvas grid**, so the in-canvas bitmap CRT font's lack of U+2191 coverage is irrelevant — top-bar buttons inherit `font-family: inherit` which resolves to the body's `ui-monospace` fallback chain via Phase 3 D-12. Verified by reading `index.html:61` (body `font-family`) + `index.html:76` (`#top-bar button` `font-family: inherit`). |
| Framework | Framework-free — vanilla DOM, native form controls, native `<dialog>`, CSS custom-property driven themes |

**Rationale:** Phase 9 inherits the entire design system from Phases 3..6.
The new affordances (top-bar button, hidden file input, drop overlay,
modal) all ride existing tokens. The single new HTML idiom (`<dialog>`)
is a native browser API requiring no library.

**Chromium-only `<dialog>` support:** PROJECT.md locks Chromium-only;
`<dialog>.showModal()` shipped in Chrome 37+ (2014). No polyfill needed.

## Spacing Scale

Phase 9 reuses Phases 3..6's established 4-px scale. All values are
multiples of 4.

| Token | Value | Usage in Phase 9 |
|-------|-------|------------------|
| xs | 4px | Top-bar `[↑ Send file]` button internal vertical padding (matches existing top-bar buttons); modal `<ul>` item vertical padding; modal `<li>` rewrite-arrow inline gap; `[data-drop-target]` overlay dashed-border dash + gap (4px on / 4px off) |
| sm | 8px | Top-bar `[↑ Send file]` button internal horizontal padding (matches existing top-bar buttons); modal `<button>` internal padding (matches Phase 5 `#connection button`); modal action-row `gap: 8px` between `[Don't send]` and `[Send N file(s)]`; `[data-drop-target]` overlay dashed border `1px` (sub-4 acceptable — see exceptions); modal `<ul>` outer margin |
| md | 16px | Modal outer padding (`16px 24px`); modal `<header>` bottom margin; modal `<footer>` top margin; vertical gap between modal sections (header / list / footer) |
| lg | 24px | Modal horizontal padding (`16px 24px`) — wider on the X-axis so long filenames don't crowd the dialog edge |
| xl | 32px | (unused in Phase 9) |
| 2xl | 48px | (unused in Phase 9) |
| 3xl | 64px | (unused in Phase 9) |

**Exceptions:**

| Exception | Value | Rationale |
|-----------|-------|-----------|
| `[data-drop-target]` overlay border width | `2px` (not multiple of 4) | A 4px dashed border on a full-canvas overlay reads as visual noise; 1px reads as accidental on HiDPI displays. 2px is the standard "drop target" weight across native OS file managers (Finder, Nautilus, Explorer) and matches user muscle memory. Treat as a single intentional sub-4 exception, locked to this one selector. |
| Modal `<ul>` `<li>` line-height | `1.5` (not 1.4) | Increased line-height on the rewrite/rejection list improves scanability when multiple files are listed; the 14-row scan-flow benefits from the extra 0.1 of vertical breathing. The chrome elsewhere stays at 1.4. Locked to `#send-modal ul li`. |
| Modal `<dialog>` `border-width` | `1px` | Matches Phase 5 `#connection` pane border treatment exactly. 1px stays sub-4 by precedent (every chrome border in the project is 1px). No exception needed — same as Phase 3..6. |
| Modal `max-width` | `560px` | Wider than `90ch` Phase 5/6 pane convention so long filenames fit on one line; 560px ≈ 70ch in the chrome monospace font. Below 90ch ceiling so the dialog never spans the full viewport on a 1280-wide screen. |
| Modal `max-height` | `60vh` | Caps modal height; if the rewrite/rejection list exceeds the cap, the `<ul>` scrolls internally (see Interaction Contracts §Modal long-list overflow). Below 80vh so the modal is always visibly a dialog, not a pseudo-page. |
| Top-bar gap between buttons | inherited 16px (`#top-bar gap: 16px`) | Phase 9 does not modify; `[↑ Send file]` slots into the existing flex-row with the same gap. |

**Pre-populated from:** Phase 3 UI-SPEC §Spacing, Phase 5 UI-SPEC §Spacing,
Phase 6 UI-SPEC §Spacing. Modal-specific values (max-width 560px,
max-height 60vh) are researcher decisions defaulted from typical filename
length distributions (CP/M 8.3 = 12 chars; original filename can run to
255 in modern OSes, but the rewrite arrow + truncated form is the
load-bearing readable element).

## Typography

Phase 9 reuses Phases 3..6's established typography. Sizes: 3 distinct
values (14px, 13px, 12px). Weights: 1 distinct value (regular 400).

| Role | Size | Weight | Line Height | Usage in Phase 9 |
|------|------|--------|-------------|------------------|
| Body | 14px | 400 | 1.4 | Top-bar `[↑ Send file]` button label (matches existing top-bar buttons); modal `<header>` title (`Sending N files via SLIDE`); modal `[Don't send]` and `[Send N file(s)]` action-row buttons; drop-overlay centered text (`Drop file(s) to send via SLIDE`) |
| Label | 13px | 400 | 1.4 | (unused in Phase 9 — modal action buttons use Body 14px to match top-bar weight; modal list items use Hint 12px to keep many-files mode scannable) |
| Hint | 12px | 400 | 1.5 | Modal `<ul>` `<li>` rewrite/rejection items (`my-doc.txt → MY-DOC.TXT` and `bad?file.txt — rejected: invalid CP/M character '?'`); modal "All files rejected — see details below" disabled-state hint |
| Display | (unused in Phase 9) | — | — | Phase 5 reserves 20px Display for polite-fail page only; not used here |

**Rationale for 12px on modal list items vs 14px on header/buttons:** the
list is the *content* (potentially many rows); header + actions are
*chrome*. 12px matches the existing chrome hint convention
(`#connection .hint`, `#debug .hint` rules in `index.html:218-222 +
375-379`) so users reading the modal experience the same density as
reading the Connection-pane error-log or hint paragraphs.

**Top-bar `↑` arrow rendering:** the `↑` glyph at 14px in the
`ui-monospace` fallback chain renders identically across CRT and Clean
themes because top-bar chrome **never uses the in-canvas bitmap font**.
Confirmed by tracing `index.html:61` (body `font-family: ui-monospace,
SFMono-Regular, Menlo, Consolas, monospace`) → `index.html:76`
(`#top-bar button { font-family: inherit; ... }`). The bitmap CRT font is
applied only to `<canvas>` glyph rendering via the canvas 2D context's
direct pixel manipulation (Phase 3 atlas pattern), never to DOM elements.
**No fallback needed for `↑`.** ASCII fallback (e.g. `[Send file]`
without arrow) is rejected.

**Pre-populated from:** Phase 3 UI-SPEC §Typography (14/12 scale), Phase
5 UI-SPEC §Typography (14/13/12 + 20px Display), Phase 6 UI-SPEC
§Typography (inherited). Phase 9 adds zero new sizes.

## Copywriting Contract

All strings below are locked. Planner and executor MUST use these exact
strings — no paraphrasing.

### Top-bar `[↑ Send file]` button (stateful — 2 states)

| State | Button label (verbatim) | Tooltip (`title` attribute, verbatim) |
|-------|-------------------------|---------------------------------------|
| enabled (idle) | `↑ Send file` | `Send file(s) to MicroBeast via SLIDE` |
| disabled (session pending or active) | `↑ Send file (sending…)` | `Transfer in progress — wait for completion` |

**Notes:**

- The `↑` is Unicode U+2191 UPWARDS ARROW. **Not** a hyphen-dash, not the
  emoji `⬆️`, not an SVG. One Unicode codepoint, rendered in the chrome
  font.
- Single space between `↑` and `Send file` for visual breathing room.
- Ellipsis is U+2026 `…`, **not** three ASCII dots `...`. Matches the
  Phase 5 `Connecting…` / `Reconnecting…` convention.
- Disabled label is `↑ Send file (sending…)` — **the arrow stays** even
  while disabled to keep the button visually identifiable; the
  parenthetical suffix is the state signal.
- No state-border-color encoding (unlike Phase 5 Connect button); state
  is encoded only in label + native `:disabled` styling (50% opacity +
  `not-allowed` cursor — inherits from existing `#top-bar
  button:disabled` pattern wherever it has been declared, or is added in
  Phase 9 if no such rule exists yet — see CSS additions).

### Drop-drop overlay

| Element | Copy (verbatim) |
|---------|----------------|
| Centered overlay text | `Drop file(s) to send via SLIDE` |

**Notes:**

- Matches REQUIREMENTS.md SLIDE-09 verbatim.
- No "or click [↑ Send file] above" supplementary copy — keeps the

### Rewrite/rejection confirm modal (`<dialog id="send-modal">`)

| Element | Copy (verbatim) |
|---------|----------------|
| Modal title (`<header>`, exactly one of these three forms based on file count) | (a) `Sending 1 file via SLIDE` (single file, no rejection) (b) `Sending 1 file via SLIDE` (single file, that file is being rewritten — same copy, the rewrite shows in the list) (c) `Sending {N} files via SLIDE` (N > 1, runtime-formatted) |
| Modal `<ul>` rewrite item | `{original} → {uppercase-truncated}` — runtime-formatted, e.g. `my-doc.txt → MY-DOC.TXT` or `REPORT-2024.csv → REPORT-2.CSV` |
| Modal `<ul>` rejection item | `{original} — rejected: invalid CP/M character '{char}'` — runtime-formatted, e.g. `bad?file.txt — rejected: invalid CP/M character '?'` |
| Modal `<ul>` rejection item (non-character reason — multiple sub-cases) | `{original} — rejected: empty filename` (leading-dot dotfile per CONTEXT D-07); `{original} — rejected: non-ASCII byte 0x{HH}` (any byte ≥ 0x80 per CONTEXT D-06); `{original} — rejected: control character 0x{HH}` (any byte < 0x20 per CONTEXT D-06) |
| Modal `[Don't send]` button label | `Don't send` |
| Modal `[Send N file(s)]` button label, all surviving (no rejection) — singular | `Send 1 file` |
| Modal `[Send N file(s)]` button label, all surviving (no rejection) — plural | `Send {N} files` (runtime-formatted) |
| Modal `[Send N file(s)]` button label, mixed (some rejected) — singular | `Send 1 file` |
| Modal `[Send N file(s)]` button label, mixed (some rejected) — plural | `Send {surviving-count} files` (runtime-formatted, count excludes rejected) |
| Modal disabled-state Send button label (all rejected — see special copy below) | `Send 0 files` — text remains "Send 0 files" but `disabled` attribute is set; visual state via 50% opacity. Spec rationale: keep the button label numerically truthful so accessibility tooling reads accurately; the disabled attribute is the affordance signal. |
| Modal all-rejected disabled-state hint (above the Don't send / Send action row) | `All files rejected — see details below.` — short single sentence, sets user expectation that the only path forward is Don't send + retry. Hidden when `surviving > 0`. |

**Plural rules:**

- Singular form: `1 file` (not `1 file(s)`).
- Plural form: `{N} files` for N = 0 or N ≥ 2.
- Title: `1 file` / `{N} files` — same plural rule as Send button.
- The title says `Sending` (gerund, present-progressive) intentionally —
  it reads as "this is what is about to happen if you click Send", not as
  a question. Matches the Phase 5 imperative-mood convention (`Connect`,
  not `Connect?`).

**Verbatim formatting examples:**

```
Sending 3 files via SLIDE

  • my-doc.txt → MY-DOC.TXT
  • REPORT-2024.csv → REPORT-2.CSV
  • bad?file.txt — rejected: invalid CP/M character '?'

```
Sending 1 file via SLIDE

  • bad?file.txt — rejected: invalid CP/M character '?'

### Copywriting invariants (inherited from Phase 5 + Phase 6, preserved here)

- **No emojis anywhere.** Pure ASCII / English (Unicode `↑` for the
  send-arrow, `→` for the rewrite arrow, `…` for ellipses, `—` for
  em-dashes, `•` for bullet points are allowed — all are typography
  punctuation, not pictorial emoji).
- **No exclamation marks.** Calm, informational tone.
- **No "please" in UI strings.**
- **No "Loading…" or "Sending…" spinners**; state is captured in the
  button label suffix `(sending…)`.
- **No modal dialogs except `<dialog>` for the rewrite/rejection confirm
  surface** — Phase 9 introduces the `<dialog>` idiom, locked to this
  one element. The Phase 5 prohibition on modals is hereby narrowed to
  "no toast / floating-banner modals"; the rewrite confirm is a
  user-initiated, two-button transactional dialog and is the appropriate
  place for native `<dialog>`.
- **Bullet character `•`** (U+2022) for the rewrite/rejection list — not
  ASCII `*` or `-`; matches modern monospace-chrome aesthetics.
- **Em-dash `—`** (U+2014) for rejection-reason separator; not ASCII
  hyphen-minus or two hyphens.
- **Right-arrow `→`** (U+2192) for the rewrite separator; not ASCII `->`.
- **All filename literals shown verbatim** — never lowercased, never
  HTML-escaped beyond what is needed for safety. Quotes `'` around the
  rejection character so the user can see whether the offender was a
  letter, a punctuation char, or whitespace. Use ASCII `'` (apostrophe),
  not curly quotes.
- **Locked verbatim strings:** the table above is the source of truth.
  No localization, no i18n. Phase 12 docs will use the same strings.

### Element ordering (DOM order below `<body>` — Phase 9 deltas only)

1. `#top-bar` — Phase 3; Phase 5 inserted `#connect-button`; Phase 6
   inserted `#clear-button`; **Phase 9 inserts `#send-file-button` +
   `#send-file-input` between `#clear-button` and `#theme-toggle`**.
2. `<details id="connection">` — Phase 5 + Phase 6 (unchanged).
3. `#terminal-wrapper` — Phase 3; Phase 6 added `#scrollback-indicator`

4. `<details id="settings">` — Phase 4 + Phase 6 (unchanged).
5. `<details id="debug">` — Phase 3 + Phase 5 (unchanged).
6. **`<dialog id="send-modal">` — NEW Phase 9, appended just before `</body>`
   so it lives at the document's modal layer.**

**Justification for `[↑ Send file]` placement (between Clear and theme-toggle):**

The top-bar button order encodes a frequency gradient — leftmost is
highest-frequency / most-state-bearing, rightmost is lowest-frequency /
display-config:

- `[Connect]` — connection state (changes 1-2x per session, but the
  state signal is glanceable continuously)
- `[Clear]` — visible-screen action (low-frequency, but a daily action)
- `[↑ Send file]` — file action (low-frequency; new in Phase 9)
- `[Clean / CRT theme]` — display toggle (rare)
- `[Green | Amber | White phosphor]` — display config (rare; visible
  only in CRT)

Sliding `[↑ Send file]` between `[Clear]` and `[Clean]` keeps the
"actions" cluster contiguous (Connect / Clear / Send file are all
verb-bearing), and the display toggles stay separated to the right.

**Justification for `<dialog>` as last document child:**

Native `<dialog>` is layered via the top-layer browser stacking context,
so DOM order does not affect z-index. Appending just before `</body>`
matches conventional modal-mounting practice and keeps the
top-of-document chrome readable.

## Interaction & State Contracts

The template's six dimensions don't fully cover an in-canvas terminal +
drop overlay + native `<dialog>` modal. The following contracts are
additive and prescriptive — executor implements them deterministically.

### Top-bar `[↑ Send file]` button state machine

| Current button state | Trigger | Next state | Side effects |
|---------------------|---------|------------|--------------|
| enabled (idle) | click | enabled | `#send-file-input.click()` programmatic — opens native file picker |
| enabled (idle) | `change` event on `#send-file-input` with non-empty `FileList` | enabled | Pass `FileList` to `file-source.js` validation pipeline; if any files survive, `<dialog>.showModal()` opens; if all files are zero entries (user cancelled the picker), no modal opens, button stays enabled |
| enabled (idle) | drag-drop `drop` event on `#terminal-wrapper` with files | enabled | Same as picker `change` — FileList → validation → modal |
| modal `[Send]` clicked | n/a (modal closes, sender path begins) | **disabled** (sending…) | Set `disabled` attribute; label changes to `↑ Send file (sending…)`; tooltip changes to `Transfer in progress — wait for completion`; auto-type `B:SLIDE R\r` fires (CONTEXT D-13); set `pendingSendSession` flag in dispatcher |
| disabled (sending…) | click | (no-op — disabled blocks clicks) | none |
| disabled (sending…) | drag-drop `drop` event on `#terminal-wrapper` | (no-op — see Drag-drop interaction during active session below) | Drop is silently rejected at the file-source layer; Phase 11 will add the "Transfer in progress — cancel first" chip (SLIDE-11). Phase 9 ships with silent rejection. |
| disabled (sending…) | dispatcher emits `EVT_SESSION_COMPLETE` | enabled | Remove `disabled` attribute; revert label + tooltip |
| disabled (sending…) | dispatcher emits `EVT_ERROR` (sender SM Error state via D-19/D-11) | enabled | Same as above; error is logged via Phase 5 error-log channel |
| disabled (sending…) | dispatcher emits `EVT_CAN` echo settled (CancelPending → reset to terminal mode per CONTEXT D-19) | enabled | Same as above |
| disabled (sending…) | tab close / page unload | (state lost — no recovery) | Phase 11 SLIDE-31 will add `visibilitychange` best-effort CTRL_CAN; Phase 9 ships without |

### Drag-drop event lifecycle on `#terminal-wrapper`

| Event | Condition | Behavior |
|-------|-----------|----------|
| `dragenter` | `e.dataTransfer.types.includes('Files')` | `e.preventDefault()`; set `[data-drop-target]="true"` on `#terminal-wrapper`; overlay becomes visible |
| `dragenter` | NOT files (text/URL/anything else) | **silent** — never `preventDefault()`, never set the attribute (CONTEXT D-04) |
| `dragover` | files | `e.preventDefault()` (required so `drop` fires); attribute already set |
| `dragover` | NOT files | silent — no preventDefault |
| `dragleave` | files | check that the leave is leaving the wrapper itself (not a child); if so, remove `[data-drop-target]`; overlay hides |
| `drop` | files | `e.preventDefault()`; remove `[data-drop-target]`; pass `e.dataTransfer.files` to file-source.js validation pipeline |
| `drop` | NOT files | silent (cannot fire if dragenter didn't preventDefault, but defense in depth) |

**`dragleave` precision:** `dragleave` fires not just on leaving the wrapper
but also when crossing into any child element. The handler MUST track
whether the leave-target is `#terminal-wrapper` itself or its descendant.
Standard pattern: increment a counter on `dragenter`, decrement on
`dragleave`; remove the attribute only when counter reaches 0.

**Drag-drop during active SLIDE session (CONTEXT D-04 + Phase 11
deferred-handoff):**

Phase 9 silently ignores drops during an active session. The drop event
handler checks `pendingSendSession !== null` OR `mode === 'send'`; if
true, the drop is silently consumed (no overlay, no modal, no chip). The
"Transfer in progress — cancel first" chip is Phase 11 SLIDE-11. Phase
9 ships with silent rejection so users don't get a confusing modal mid-send.

### Native `<dialog>` interaction contract (CONTEXT D-05)

| Trigger | Behavior |
|---------|----------|
| `<dialog>.showModal()` | Modal opens centered viewport-overlay; backdrop dims rest of page; focus trapped inside dialog (native behavior) |
| Initial focus on open | **`#send-modal-cancel` button** (the safer of the two — pressing Enter on a freshly-opened modal should NOT trigger a destructive-feeling Send action). User Tab-advances to Send button; Tab-cycles wrap. |
| Tab order inside modal | (1) `#send-modal-cancel` → (2) `#send-modal-send` → (back to Don't send via Tab-cycle wrap or Shift+Tab). The `<ul>` items are NOT tabbable (read-only content). |
| Esc key | Native `<dialog>` Esc-to-cancel behavior. Fires `cancel` event then `close` event. Phase 9 attaches a `cancel` listener that simply re-enables the top-bar button (since clicking Don't send and pressing Esc are the same outcome). |
| Click outside modal (on `<dialog>::backdrop`) | **Closes the modal** (treated as Cancel). Standard click handler: `dialog.addEventListener('click', e => { if (e.target === dialog) dialog.close('cancel'); })` — works because clicks on the backdrop pass through to the dialog itself (native browser behavior). |
| Click `[Don't send]` | `dialog.close('cancel')` — modal closes, no SLIDE session opens, top-bar button stays enabled, `FileList` discarded |
| Click `[Send N file(s)]` (when surviving > 0) | `dialog.close('send')` — modal closes; surviving files are packed into metadata via `packSendMetadata`; `enterSendMode({ files })` called; auto-type fires; top-bar button transitions to disabled |
| Click `[Send 0 files]` (disabled — all rejected) | (no-op — disabled attribute blocks click) |
| Modal long-list overflow (rejection list > 60vh) | Internal scroll on `#send-modal ul` (CSS `overflow-y: auto` on the `<ul>`). The `<header>` and `<footer>` stay pinned via flex-column layout. Smooth-scroll behavior is browser-default; no custom polyfill. |

**Reduced-motion accessibility:** Phase 9 ships with **zero CSS
animations or transitions** on the new surfaces. The drop overlay
`display: flex` toggles instantly (no opacity transition); the modal
opens via native `<dialog>` semantics (no fade-in). This is consistent
with the project's existing motion-light posture (the only animations
in the project are the 100ms bell flash and the cursor blink). No
`prefers-reduced-motion` rule is needed because there is no motion to
reduce.

### Auto-type interaction (CONTEXT D-13)

| Step | Action | UI surface |
|------|--------|-----------|
| 1 | User clicks `[Send N file(s)]` in modal | Modal closes |
| 2 | `pushTxBytes(textEncoder.encode('B:SLIDE R\r'))` | Bytes flow through tx-sink owner-gate (`'terminal'` at this moment); writer transmits over Web Serial |
| 3 | Auto-typed bytes echo back from MicroBeast | **Visible doubling in terminal canvas** — Phase 9 ships with this, Phase 11 SLIDE-14 swallow-echo filter fixes it |
| 4 | `pendingSendSession` flag set | No UI surface change yet; top-bar button transitions to disabled now |
| 5 | Z80 emits `ESC ^ S L I D E` wakeup | Dispatcher transitions to `'send'` mode (D-13 wakeup-completion clause) |
| 6 | Sender SM exchanges RDY / ACK / data frames | No UI; introspection via `window.__slide` per CONTEXT D-18 |
| 7 | Sender SM emits `EVT_SESSION_COMPLETE` | Top-bar button transitions back to enabled |

**Visible doubling acknowledged:** The auto-typed `B:SLIDE R\r` and the
Z80's local echo of the same characters render twice in the terminal
canvas. This is intentional per CONTEXT D-13 / Phase 11 deferral. Phase
9 manual UAT may flag this as jarring; pull-forward is at planner
discretion per CONTEXT D-13's escape-hatch clause. The UI-SPEC contract
is "doubling is acceptable for Phase 9".

### Modal long-list overflow

| Surviving + rejected count | Behavior |
|----------------------------|----------|
| ≤ ~12 files (fits 60vh on 720px viewport) | Modal renders without scroll; `<ul>` natural height |
| > 12 files | Modal `<ul>` scrolls internally via `overflow-y: auto`; header + footer stay pinned via flex-column layout (`#send-modal { display: flex; flex-direction: column; max-height: 60vh; }` + `#send-modal ul { flex: 1 1 auto; min-height: 0; overflow-y: auto; }`) |
| Filename collision detection (Phase 12 SLIDE-36) | Out of scope for Phase 9. The modal lists rewritten names verbatim; if two files truncate to the same 8.3 name, they both appear in the list with no collision warning. Phase 12 will add collision detection + auto-rename UX. |

### Focus retention on modal close

When the modal closes (any path — Cancel, Esc, click-backdrop, Send),
focus MUST return to a deterministic location:

| Modal close path | Focus restored to |
|------------------|-------------------|
| Cancel / Esc / click-backdrop | `#send-file-button` (the trigger button) — user can re-attempt without re-clicking |
| Send (success) | `#terminal-wrapper` — the active surface during a session is the terminal; user can keep typing (though keystrokes are owner-gated to `'slide'` so they no-op until session ends) |

Implementation: `dialog.addEventListener('close', e => { if
(dialog.returnValue === 'send') terminalWrapper.focus(); else
sendFileButton.focus(); })`.

### Focus retention on top-bar button click (extends Phase 4 D-16)

The new `[↑ Send file]` button MUST follow the existing top-bar
mousedown-preventDefault pattern (Phase 4 D-16) so clicking the button
does not steal focus from `#terminal-wrapper`.

| Trigger | Element | Behaviour |
|---------|---------|-----------|
| `mousedown` | `#send-file-button` | `e.preventDefault()` to retain terminal focus (matches Phase 4 D-16 + Phase 6 `#clear-button` pattern); click handler fires normally and opens the picker |
| `mousedown` | `#send-modal-cancel`, `#send-modal-send` | NO preventDefault — modal owns focus while open; native `<dialog>` focus semantics prevail. Modal close handler restores focus per the table above. |

### Drag-drop vs pointer-select coexistence (Phase 12 forward-reference)

The `#drop-overlay` has `pointer-events: none` so the underlying
`#terminal-wrapper`'s pointer-select (Phase 6 SESS-02) keeps working
when no drag is in progress. The drag-drop vs pointer-select isolation
regression spec is **Phase 12 SLIDE-12** — Phase 9 ships the
`pointer-events: none` rule but does NOT add a regression test;
Phase 12 covers that.

### Keyboard shortcuts introduced

Phase 9 introduces **zero new keyboard shortcuts** (CONTEXT D-02 lock).
Send is mouse-only via the top-bar button or drag-drop. Cancel-while-
sending is Phase 10 (Esc disambiguation chain, slot 2 of 4 — SLIDE-27).
Modal Esc-to-close is native `<dialog>` behavior, not a new app-level
chord.

### Live announcement for screen readers

The top-bar button label change (`↑ Send file` → `↑ Send file
(sending…)`) is announced because `<button>` text changes naturally
trigger live-region behavior in screen readers when the button has
focus. Phase 9 does NOT add `aria-live` to the button. The modal
heading + list are statically rendered before `showModal()`, so screen
readers read them normally. Modal `<header>` h2 is associated via
`aria-labelledby` on `<dialog>` for the dialog's accessible name.

### Semantic markup

- `<button>` for `#send-file-button`, `#send-modal-cancel`,
  `#send-modal-send` — native interactive elements.
- `<input type="file" multiple hidden>` for `#send-file-input` — native
  file picker; `hidden` attribute ensures it's not in the tab order.
  Programmatic `.click()` opens the picker.
- `<dialog>` — native modal element with built-in focus trap, Esc-to-
  close, backdrop dimming. No ARIA `role="dialog"` needed (native
  `<dialog>` has implicit role).
- `aria-labelledby="send-modal-title"` on `<dialog>` — gives the dialog
  its accessible name (the `<h2>` text inside `<header>`).
- `<ul>` + `<li>` for the rewrite/rejection list — native list
  semantics; screen readers announce "list of N items" on entry.
- `<header>` + `<footer>` inside `<dialog>` — semantic landmark elements
  for the dialog's structural regions.
- `aria-hidden="true"` on `#drop-overlay` — overlay is visual feedback
  only; the screen-reader path is via the top-bar button + native
  drag-drop announcements. Inner spans wrapping `•` and `→` use
  `aria-hidden="true"` so the bullet/arrow characters aren't announced
  redundantly.

### `<dialog>` accessible name + description

```html
<dialog id="send-modal" aria-labelledby="send-modal-title">
  <header>
    <h2 id="send-modal-title">Sending 3 files via SLIDE</h2>
  </header>
  …
</dialog>
```

The dialog's accessible name comes from `<h2>` text via
`aria-labelledby`. No `aria-describedby` on the dialog itself; the `<ul>`
content is read by the screen reader as a list when focus enters it (or
on dialog open in some screen readers). If Plan 09 manual UAT shows
screen reader users need an explicit description, an `aria-describedby`
pointing at a hidden description span can be added incrementally.

### Drop-overlay screen reader path

The drop overlay is `aria-hidden="true"` because:

1. The drop interaction is initiated by the user (a drag), so the
   visual feedback is a confirmation of an in-progress user gesture, not
   a state announcement.
2. Drag-and-drop on the file system is a well-established native
   interaction; screen readers handle it via OS-level conventions.
3. The alternative path (clicking `[↑ Send file]`) is fully
   keyboard-accessible and screen-reader-described via the button label
   + tooltip.

This is intentional asymmetry — the drag-drop path is a
mouse-first/touchpad-first affordance; the click path is the
keyboard/screen-reader path. Both end in the same modal.

### Color-only state encoding check

The drop overlay uses **both** color (10% accent tint) AND shape (2px
dashed border) AND text (`Drop file(s) to send via SLIDE`) — three
redundant signals. The top-bar button uses **both** label text (`↑ Send
file (sending…)`) AND opacity (50% disabled) — two redundant signals.
WCAG 1.4.1 (use of color) is satisfied at every Phase 9 surface.

### Screen reader note (inherited)

PROJECT.md Out-of-Scope lists "Screen-reader support on canvas" as
deferred. Phase 9 chrome (top-bar button, drop-overlay text, modal) IS
accessible. The in-canvas terminal content during a SLIDE session is
not. This asymmetry is documented and intentional.

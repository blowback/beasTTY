# Phase 12 — UI Design Contract (DELTA)

> Visual and interaction contract for Phase 12 (SLIDE UX Polish, Docs &
> Real-Hardware UAT). Generated by gsd-ui-researcher; verified by
> gsd-ui-checker.
>
> **This is a DELTA UI-SPEC.** Phases 9 and 11 already locked the entire
> design system surface this phase touches. Phase 12 only specifies the
> *additions* and *exceptions* needed for three UI-touching deliverables:
>
> 1. **SLIDE-36** — extends the Phase 9 send `<dialog>` modal with one new
>    row kind (`'collision'`), a three-action button row replacing the
>    existing two-button footer when collisions are present, and a
>    default-focus override scoped to the collision-present mode.
> 2. **SLIDE-12** — *no visual delta*. Pure interaction contract:
>    `selection.js:onPointerDown` early-returns when the drop overlay is
>    active. Documented here for completeness (predicate mechanism +
>    in-flight drag policy + post-drop selection-clear contract).
> 3. **SLIDE-38** — extends the Phase 11 chip lifecycle with one new
>    transient state (`first-use-confirm`), adds a Settings input
>    rejection visual (orange-free reuse of muted/error tokens), and
>    pins the auto-send safety regex.
>
> **Inherited verbatim from Phase 9 (`09-UI-SPEC.md`):** entire send
> modal contract — `<dialog id="send-modal">` DOM, focus trap, click-
> outside-to-dismiss, `<ul>` list with `'rewrite'` / `'unchanged'` /
> `'rejected'` row kinds, `[Don't send]` / `[Send N files]` button row,
> `cancelBtnRef` default focus, all copy strings (`Don't send`, `Send
> {N} files`, `All files rejected — see details below.`, modal title
> `Sending {N} files via SLIDE`), `<dialog>::backdrop rgba(0,0,0,0.65)`,
> `max-width: 560px`, `max-height: 60vh`, modal `<li>` 12px Hint @ 1.5
> line-height (sub-4 exception), drop-overlay 2px dashed border (sub-4
> exception), tint `color-mix` + RGBA fallback per phosphor.
>
> **Inherited verbatim from Phase 11 (`11-UI-SPEC.md`):** entire SLIDE
> chip contract — `#slide-chip` placement (`bottom: 8px; left: 8px`),
> z-index 5, lifecycle states (`hidden` / `awaiting-wakeup` /
> `awaiting-timeout` / `active` / `cancelled-summary` / `sent-summary`
> / `received-summary` / `error` / `drop-rejected-flash`), state-change
> observer + 250 ms refresh tick, inline button bracketing pattern
> (`[Cancel]` / `[Retry]` / `[Force start]`), 12px Hint typography,
> `aria-live="polite"`, two-space token separators, `rgba(0,0,0,0.65)`
> backdrop, theme-aware accent border, the entire Settings SLIDE
> sub-block (`<details class="reserved" id="settings-slide">` with the
> 4 rows including auto-send `<input type="text">` + Compatibility
> `<select>` + show-summary checkbox + Save-to-folder), all SLIDE
> sub-block copy strings.
>
> **Hard architectural constraint:** Phase 12 is JS / HTML / CSS /
> Markdown only. **Zero Rust changes** (mirrors Phase 11 invariant).
> **Zero new color hex values, zero new typography sizes, zero new
> font families, zero new spacing tokens, zero new third-party
> dependencies.** Every Phase 12 surface rides existing Phase 3..11
> tokens.

### Destructive policy (Phase 12 reaffirmed)

**Phase 12 introduces zero destructive UI surfaces.** Every Phase 12
control is non-destructive:

| Control | Why non-destructive |
|---------|---------------------|
| `[Send N renamed]` | Adds `~N` suffixes; no file is dropped or renamed on disk; the rewrite is inside the SLIDE wire payload only. Auto-rename is the *safe* path (CONTEXT D-03). |
| `[Send only first]` | Drops files from the *current send batch* only; no on-disk mutation; the user can retry with a different selection. |
| `[Refuse batch]` | Closes modal, drops the in-memory batch only; matches Phase 9 `[Don't send]` semantics — non-destructive. |
| `[Cancel]` (modal) | Phase 9 inheritance — non-destructive. |
| First-use-confirm chip `[Confirm]` | User affirmation; no destructive side-effect. |
| First-use-confirm chip `[Reset to default]` | Restores DEFAULTS literal (`'B:SLIDE R\r'`); user *opted into* the change earlier and is opting out now — explicitly non-destructive recovery. |
| Settings auto-send `<input>` invalid-value border | Visual cue only; `savePrefs` still persists the value (per RESEARCH §Pitfall: "Blocking save when the auto-send regex fails" → save-time blocking is forbidden); use-time validation in `slide.js` is the hard gate. The input border-color shift is a non-destructive hint, not a state-color encoding. |

**Therefore Phase 12 declares zero destructive token.** No red `#e04040`,
no orange warning. The "rejection" visual on the auto-send input uses
**muted-color border** (`rgba(255,255,255,0.6)`) plus an **inline hint
below the input**, following the Phase 6 D-35 reset-prefs inline-confirm
pattern (visual cue + hint text, no destructive coloring).

**Rationale:** Per Phase 9 §Color: "Destructive: not used in Phase 9.
No destructive actions." Per Phase 11 §Color: "Phase 11 has zero
destructive UI surfaces." This invariant is preserved in Phase 12.
Adding a destructive red specifically for `<input>` validation would
break the project's "calm, informational tone" copywriting invariant
and add a hex value for one selector.

### First-use-confirm chip copy (NEW Phase 12, SLIDE-38)

The chip surface for first-use confirmation reuses Phase 11's chip
lifecycle pattern — adds **one new lifecycle state** (`first-use-confirm`)
with bracketed inline buttons in the same idiom as
`awaiting-timeout`'s `[Retry] [Cancel] [Force start]` row.

| Element | Copy (verbatim) | Notes |
|---------|------------------|-------|
| Chip — `first-use-confirm` state body | `Confirm auto-send: {VALUE}  [Confirm]  [Reset to default]` | `{VALUE}` is the user's typed value with `\r` rendered as the literal two-character sequence `\r` (backslash + r — NOT a CR control character; NOT escaped to a Unicode \r0D pictograph). Example: `Confirm auto-send: B:DIR\r  [Confirm]  [Reset to default]`. Two-space separators per Phase 11 token-separator convention. |
| Chip — `first-use-confirm` aria-label | `Confirm auto-send command before first use` | Polite live-region announcement. |
| Settings auto-send validation hint (NEW row inside `#slide-auto-send-row`) | `Auto-send command unsafe — using disabled.` | Em-dash `—` (U+2014). Trailing period. Hint typography (12px Hint, muted color). Renders **only when** the use-time `SAFE_AUTO_SEND_RE` test fails AND the user has clicked Send-file (i.e. validation is use-time, not save-time, per RESEARCH §Anti-Patterns + Pitfall 5). The hint persists in the Settings row until the user edits the input back to a safe value. |

**Chip lifecycle integration (locked):**

| Trigger | Chip state | Visible |
|---------|-----------|---------|
| User clicks `↑ Send file`; `prefs.slideAutoSendCommand` is non-empty AND has changed since last confirmation (per CONTEXT D-* "exact-match flag, keyed to the exact string") | `first-use-confirm` | yes — replaces any prior chip content; chip shows confirm/reset row |
| User clicks `[Confirm]` | `awaiting-wakeup` | yes — auto-type fires, normal Phase 11 lifecycle resumes |
| User clicks `[Reset to default]` | `hidden` | no — auto-send command is reset to DEFAULTS literal `'B:SLIDE R\r'`, `slideAutoSendCommandConfirmed` flag set to true, chip auto-hides; user must click `↑ Send file` again to actually send |

**Pitfall mitigation (RESEARCH §Pitfall 6):** First-use-confirm
fires **only at session START** (i.e. within `enterSendMode` BEFORE
auto-type bytes go on the wire), never during a Settings input change
event. This guarantees the chip never overwrites an in-flight active-
session chip.

## Layout Contract

### Where Phase 12 ADDS markup (delta only)

#### A. Modal `<li class="collision">` row shape (NEW)

Runtime-built by `file-source.js:processFiles` as a fourth `<li>` row
kind, alongside the existing `'rewrite'` / `'unchanged'` / `'rejected'`
kinds.

```html
<!-- Collision (will be sent with ~N suffixes — one entry per collision group): -->
<li class="collision">
  <span aria-hidden="true">•</span>
  <span class="orig">REPORT.TXT</span>
  <div class="rename-list" aria-label="Renamed to: REPORT.TXT, REPORT~1.TXT, REPORT~2.TXT">
    <span aria-hidden="true">↳</span>
    REPORT.TXT, REPORT~1.TXT, REPORT~2.TXT
  </div>
</li>
```

The bullet `•` and `↳` marker are wrapped in `aria-hidden="true"` spans
(matches Phase 9 `<li class="rewrite">` and `<li class="rejected">`
patterns). The full rename list is announced via `aria-label` on the
`.rename-list` div so screen-reader users hear "Renamed to: REPORT.TXT,
REPORT~1.TXT, REPORT~2.TXT" once per row.

**Why a `<div>` for the rename list (not a `<span>`):** the list needs
to be a block-level child for `margin-left: 24px` indent + line wrap
behavior. Phase 9's `<li>` uses inline-flex layout for single-line items;
the collision row breaks to a two-line shape. The CSS extension below
specifies the override.

#### B. Modal three-action button row (NEW — replaces the two-button row when collisions are present)

```html
<!-- Existing Phase 9 two-button row (still used when no collisions): -->
<footer>
  <button id="send-modal-cancel" type="button">Don't send</button>
  <button id="send-modal-send" type="button">Send {N} files</button>
</footer>

<!-- NEW Phase 12 three-button row (used when collisions present): -->
<footer>
  <button id="send-modal-send-renamed" type="button" value="send">Send {N} renamed</button>
  <button id="send-modal-first-only" type="button" value="first-only">Send only first</button>
  <button id="send-modal-refuse" type="button" value="refuse">Refuse batch</button>
</footer>
```

The two footer variants share the same `<footer>` styling (Phase 9 verbatim — `margin: 16px 0 0 0; padding-top: 8px; border-top: 1px solid var(--chrome-border); display: flex; justify-content: flex-end; gap: 8px`). The `gap: 8px` accommodates the third button without modification.

**Implementation note:** the planner picks whether to reuse the same `<footer>` element with runtime swap of inner buttons, OR maintain two separate `<footer>` elements toggled via `hidden`. Either approach satisfies this contract. **Researcher default: single `<footer>`, runtime swap of inner buttons** — minimizes DOM, matches Phase 9 modal-flow pattern.

The `id` attributes are advisory; Phase 9 uses `id` selectors so the planner is encouraged to keep the convention. The `value` attribute on each button is the source of truth for `<dialog>.close()` `returnValue` per Phase 9 D-05 + RESEARCH Pattern 4.

#### C. First-use-confirm chip lifecycle state (NEW)

No new DOM. Reuses Phase 11's `<button id="slide-chip">` outer + `<span id="slide-chip-text">` inner. New lifecycle state appended to the Phase 11 state machine:

| State | Hidden? | Chip text | aria-label | Auto-hide |
|-------|---------|-----------|------------|-----------|
| `first-use-confirm` (NEW Phase 12) | no | `Confirm auto-send: {VALUE}  [Confirm]  [Reset to default]` | `Confirm auto-send command before first use` | **no auto-hide** — waits for user click on either button. If user dismisses via Esc or clicks elsewhere, chip auto-hides after **30 s** (defensive timeout — chip should not pin indefinitely; safer-default per Phase 11 chip-state pattern). |

### SLIDE-12 — Pointer/drop isolation (NO VISUAL DELTA)

**Behavioral contract only.** No new DOM, no new CSS, no new copy.

**Predicate mechanism (locked, RESEARCH-default):**

`selection.js:onPointerDown` reads the existing `[data-drop-target]`
attribute on `#terminal-wrapper` directly. Matches the established
`[data-focused]` (Phase 3) and `[data-scrolled-back]` (Phase 6)
cross-module patterns. **No new DOM signal, no new injected predicate.**

```js
// www/input/selection.js — Phase 12 SLIDE-12 insertion at line 113
function onPointerDown(ev) {
    if (ev.button !== 0) return;
    // SLIDE-12: drop overlay active → defer to file-source.js drag handlers.
    // canvasRef.parentElement is #terminal-wrapper (the drop attribute owner).
    // Strict equality on the literal string 'true' — matches Pitfall 4.
    if (canvasRef.parentElement?.getAttribute('data-drop-target') === 'true') {
        return;
    }
    ev.preventDefault();
    // ... existing body unchanged
}
```

**In-flight drag policy (locked, RESEARCH-default):**

If a pointer-drag is already in flight when `[data-drop-target]` activates
(file enters the wrapper mid-drag), `selection.js` continues the in-flight
drag normally — the early-return only affects NEW pointerdown events,
not in-flight gestures. **The post-drop selection-clear contract handles
the cleanup:**

**Post-drop selection-clear contract (locked):**

After a successful `drop` event in `file-source.js` (i.e. the SLIDE
session opens), `file-source.js` calls `selection.clearSelection()` (or
the equivalent via injected hook) so any partial pointer-select
artefact left from a half-completed drag is wiped. This is the SLIDE-12
SC#1 success criterion: "no ghost selection / inverse-text artefact
remains after a drop".

**Implementation seam:** `wireSelection({ ..., clearSelection })`
already exposes a clearSelection-like API (Phase 6). `file-source.js`
gains a `clearSelectionFn` injection via `wireFileSource({ ...,
clearSelectionFn })` and calls it after the existing `setDropTarget(false)`
in `onDrop`. **Planner discretion** on whether to use a direct module
import (no new wiring), an injected callback (matches existing Phase 11
chipApi/cancelSlideRecvLazy pattern), or a `selectionchange`-style
event subscription. Researcher default: **injected callback** —
matches Phase 11 boot-wiring discipline (`wireSlideChip` ↔
`wireFileSource` ↔ `wireSlideDispatcher` opts plumbing).

**Drag-in-flight + drop-overlay-activates-mid-drag corner case
(researcher-default per CONTEXT Claude's Discretion):**

> "Default: if a drag is in flight when drop overlay activates, cancel
> the drag (`clearSelection()`). Drop should win because file-source
> already set the overlay attribute by the time `dragenter` fired."

This is a behavioral assertion only — no UI signal. The
`clearSelection()` call is fire-and-forget; the user sees their partial
selection vanish at the moment the drop overlay appears. Acceptable
because the user explicitly initiated a drop gesture, which supersedes
their intent to continue selecting.

**Test contract (the SLIDE-12 Playwright spec):**

| Test name | Assertion |
|-----------|-----------|
| `pointerdown does not start selection while drop overlay active` | After programmatically setting `[data-drop-target]="true"` on `#terminal-wrapper`, simulate `pointerdown` on canvas → assert no selection bounds in `__selectionState`, no inverse-text artefact in canvas pixel readout. |
| `pointerdown starts selection normally when drop overlay inactive` | Without `[data-drop-target]` set, `pointerdown` → drag → release → assert selection bounds present + copyable text matches expected substring (Phase 6 regression). |
| `drop event clears any in-flight pointer-select bounds` | Simulate `pointerdown` + `pointermove` (selection in flight); fire synthetic `dragenter` with files; fire `drop` → assert `__selectionState` cleared, no inverse-text artefact remains. |

### SLIDE-38 — First-use-confirm chip & Settings input flow (DELTA)

**First-use detection rule (locked, CONTEXT Claude's Discretion default):**

```js
// In slide.js readAutoSendCommandBytes (or a dedicated helper),
// after the safety regex check:
const cmd = prefsRef.slideAutoSendCommand;       // e.g. 'B:DIR\r' (custom)
const confirmedFor = prefsRef.slideAutoSendCommandConfirmed;  // string keyed to last-confirmed value
if (cmd === '') return null;                     // SLIDE-13 disabled sentinel
if (!SAFE_AUTO_SEND_RE.test(cmd)) {
    showSettingsValidationHint();                // unhide the validation hint
    return null;                                 // hard gate — auto-send disabled
}
if (cmd !== 'B:SLIDE R\r' && cmd !== confirmedFor) {
    // First use of this exact custom string — surface the chip.
    return { needsConfirmation: true, cmd };
}
return { cmd };                                  // happy path
```

`SAFE_AUTO_SEND_RE` is locked at:

```js
const SAFE_AUTO_SEND_RE = /^[A-Za-z0-9:]*\r$/;
// Empty string short-circuits BEFORE the regex (SLIDE-13 disabled sentinel).
```

**Test contract (locked from RESEARCH §Pitfall 5 + CONTEXT specifics):**

| Input | Pass? | Reason |
|-------|-------|--------|
| `''` | yes | SLIDE-13 disabled sentinel — bypasses regex |
| `'B:SLIDE R\r'` | yes | default value |
| `'A:SLIDE R\r'` | yes | drive switch |
| `'B:SLIDE\r'` | yes | no R; user drives slide.com manually |
| `'B:DIR\r'` | yes | valid CP/M command (would trigger first-use-confirm chip if not previously confirmed) |
| `'\r'` | yes | bare CR — `*` not `+` admits empty body |
| `'B:SLIDE R'` | NO | missing trailing `\r` |
| `'B:SLIDE R\n'` | NO | LF instead of CR |
| `'B:SLIDE R; rm -rf /\r'` | NO | semicolon |
| `'B:SLIDE R\rB:DIR\r'` | NO | multiple `\r` |
| `'B:SLIDE\x07R\r'` | NO | control character (BEL) |
| `'B:SLIDE\\R\r'` | NO | backslash |

**Settings input invalid-state visual lifecycle (locked):**

| Trigger | `data-invalid` attribute | `.validation-hint` `[hidden]` |
|---------|---------------------------|-------------------------------|
| User types invalid value (e.g. `B:SLIDE R; rm -rf /`) into `<input>` | (no change yet — save-time validation does NOT immediately set `data-invalid`) | (no change — hint stays hidden) |
| `savePrefs` runs (250 ms debounce) and detects unsafe value | `data-invalid="true"` | (no change — hint stays hidden until use-time gate fires) |
| User clicks `↑ Send file` and `readAutoSendCommandBytes` returns `null` | `data-invalid="true"` (already set) | unhide |
| User edits `<input>` to a safe value | `data-invalid` removed | hide |

**Why save-time sets `data-invalid` but not the hint:** The visual border-shift is a passive cue that does not interrupt user typing. The hint text `Auto-send command unsafe — using disabled.` is a strong signal and only surfaces after the user *attempts to use* the broken command. This matches RESEARCH §Pitfall + Anti-Patterns: "Validation at use, not at save."

**First-use-confirm chip exclusivity (locked, RESEARCH §Pitfall 6):**

The first-use-confirm chip surfaces ONLY at session START (i.e. inside
`enterSendMode` BEFORE `pushTxBytes` fires). It NEVER surfaces during a
Settings input change event, NEVER overlays an active-session chip,
NEVER overlays an `awaiting-wakeup` / `awaiting-timeout` chip. If the
user changes `slideAutoSendCommand` mid-session, the change persists
silently in prefs; the chip surfaces only on the NEXT session start.

Purpose: The instrumented hardware capture (logged in `12-HUMAN-UAT.md` Gap 2 root_cause) and slide-team consultation confirmed that Phase 5 D-09/D-11 (de-assert RTS on connect to avoid CP2102N reset-pulse via DTR/RTS-wired GPIOs) is incorrect for the MicroBeast: RTS goes to the Z80's CTS pin and Z80-side UART hardware auto-flow-control blocks ALL Z80 transmits when host RTS=low. The Phase 9/10/11 send path is structurally correct; the bytes never leave the Z80 because the Z80 UART won't transmit. Asserting RTS on connect unblocks every Z80→host transmit (wakeup byte, ACKs, file data) for patched-firmware MicroBeasts.

The toggle preserves the safe-default-RTS-low option for users on hardware where the original Pitfall #12 reset concern applies (e.g. clones with RTS wired to a GPIO that resets the Z80). DTR remains de-asserted on connect in ALL paths — DTR-as-reset is more credible than RTS-as-reset and the slide-team request was RTS-only.

## Known traps for the executor

1. **Browser focus-visible behavior is non-trivial.** The regression test for Gap 1 MUST use the same trigger path as production — programmatic `.focus()` after a pointer-initiated event (drop / file-picker `change`). Synthetic `page.keyboard.press('Tab')` would activate `:focus-visible` and the test would FALSELY pass even if the data-focused attribute fix were absent. Use `setInputFiles` on `#send-file-input` to fire the production `change` handler. Note further: under Playwright's synthetic file-input path, Chromium MAY still classify the resulting `.focus()` as keyboard-like and match `:focus-visible` — meaning the computed-color assertion alone could pass even if the JS attribute write were absent. The `data-focused === 'true'` attribute poll is therefore the load-bearing regression guard; the computed-color check is defense-in-depth.

2. **CSS specificity arithmetic — source-order is a TRAP.** The Gap 2 fix uses `[data-theme] #settings-slide #slide-auto-send-input[data-invalid="true"]` with specificity (0,2,2,0). This beats:
   - the base rule `#settings-slide #slide-auto-send-input` at line 217: (0,2,0,0) ✓ wins by 2 attributes
   - the `:focus-visible` rule `#settings-slide #slide-auto-send-input:focus-visible` at line 227: (0,2,1,0) ✓ wins by 1 attribute
   **Why not just (0,2,1,0)?** It would tie with the `:focus-visible` rule. The cascade falls back to source-order. A future author moving the invalid rule above line 227, splitting the file, or refactoring stylesheet ordering silently regresses the focused-and-invalid case. (0,2,2,0) wins regardless of order — the contract is enforced by specificity arithmetic, not by file layout. Document this in the rule's comment block (verbatim text provided in Task 2's action).

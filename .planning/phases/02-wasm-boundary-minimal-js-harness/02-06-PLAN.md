<!-- NEW pure-Rust contracts you will ADD in Task 1 -->

```rust
impl Terminal {
    /// Like `feed` but does NOT clear or take `host_reply`. Bytes accumulate
    /// into `self.host_reply` (preserved across calls until JS reads + clears).
    /// Allocates only when `host_reply` needs to grow past its current capacity.
    /// Pre-reserves a small capacity at construction so the empty-reply common
    /// case is zero-alloc post-init.
    pub fn feed_silent(&mut self, bytes: &[u8]);

    /// Pointer into the host_reply buffer — stable across `feed_silent()`
    /// calls UNTIL `host_reply` is grown (rare; only when an ESC Z fires
    /// against a host_reply that has been previously truncated below 3 bytes
    /// of capacity, which the constructor pre-reserve prevents). JS must
    /// re-derive the Uint8Array view if `wasm.memory.buffer` is replaced
    /// OR if the pointer changes between calls. Mirrors the D-03 contract.
    pub fn host_reply_ptr(&self) -> *const u8;

    /// Length in bytes of the currently-pending host reply. 0 in the common
    /// case (no ESC Z in the most recent feed). Constant-time field read.
    pub fn host_reply_len(&self) -> usize;

    /// JS calls this AFTER reading the reply view. Sets len=0 but preserves
    /// capacity, so the next ESC Z reuses the same allocation.
    pub fn clear_host_reply(&mut self);
}
```

<decision_reversal_acknowledgement>
This plan deliberately reverses Plan 04's "reDeriveViews on every render tick"
decision (recorded in 02-04-SUMMARY.md `key-decisions`):

> "reDeriveViews() inside every render tick: chose correctness-over-performance.
> The two Uint8Array constructors cost ~microseconds; the alternative
> (derive-once + re-derive-on-resize) adds an invariant Phase 3/4/5 must
> remember. Phase 3 can benchmark and back off if profiler shows it matters
> (it won't at 80x24 @ 60 Hz)."

The prediction "it won't matter at 80x24 @ 60 Hz" was tested by SC-3's profiler
demonstration and DISAGREED — the user observed a visible sawtooth. The reversal
is therefore evidence-driven, not a blind override. The new shape uses a
`cachedBuffer` guard that costs ONE identity comparison per render and zero
allocations in the common case, and the "invariant Phase 3/4/5 must remember"
shrinks to: "if you call `term.resize()`, the next `reDeriveViews()` will
detect the buffer swap automatically." That is checked at the call site (single
identity compare) rather than maintained as out-of-band knowledge — strictly
better than Plan 04's framing.

The Rust signature change (Source #1 in the gap diagnosis) is the load-bearing
move — the dominant per-click allocation is the wasm-bindgen-generated `.slice()`
which is invisible from www/main.js but present on every feed() call regardless
of whether the host_reply is empty (the common case in Phase 2 since only
ESC Z emits a reply). Source #2 (reDeriveViews) is the visible-in-source one,
but Source #1 is the bigger byte count and the reason the user's profiler
showed a sawtooth even with simple ASCII input that produces no host_reply.
</decision_reversal_acknowledgement>
